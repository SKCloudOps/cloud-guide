## Architecture & Design

### Q1. What is the fundamental difference between High Availability and Disaster Recovery?  `⚡ High Freq`

Most candidates conflate these two. The distinction is operational continuity vs. existential survival . High Availability (HA) Eliminates single points of failure within a running system. Goal: maximize uptime (99.9–99.999%). Operates within one region. Disaster Recovery (DR) Restores a system after a catastrophic failure that HA cannot prevent — region outage, data corruption, ransomware. Goal: minimize RTO/RPO. "HA and DR solve different threat models. HA handles component failure — a node dies, traffic shifts to another. I achieve that with multi-AZ EKS, ALB health checks, and RDS Multi-AZ. But HA cannot protect against an entire AZ or region going dark, or an operator running DROP TABLE on prod. That's DR territory — a deliberate, tested runbook to restore service within agreed RTO/RPO bounds. At Takeda, I separated them explicitly: HA is in the IaC config, DR is a quarterly tested runbook with a pilot-light environment in us-west-2. You need both — HA keeps the lights on daily, DR keeps the business alive after black-swan events." Saying "HA is when you have multiple servers" with no mention of AZ spread, failover mechanics, or the distinction between MTTR and RPO.

### Q3. Define RTO and RPO. Compare all 4 DR strategies.  `◈ Deep` `⚡ High Freq`

RTO = Recovery Time Objective — maximum acceptable downtime. RPO = Recovery Point Objective — maximum acceptable data loss measured in time. Lower = more expensive. Strategy RTO RPO Cost How it works Backup & Restore Hours–Days Hours–Days $ S3 snapshots, RDS backups. Restore from scratch on disaster. Cheapest, slowest. Pilot Light 10–30 min Minutes $$ Core DB replicating to DR region. Minimal compute off. On disaster, scale up compute and re-point DNS. Warm Standby Minutes Seconds–Minutes $$$ Scaled-down but running copy in DR region. Handles low traffic. Scale to full on disaster. Multi-Site Active-Active ~0 (near-zero) ~0 $$$$ Both regions serve production traffic. Route 53 health routing splits load. Global Aurora or DynamoDB Global Tables for data. Failover = DNS weight change. "The strategy you pick isn't just about RTO/RPO — it's about the business cost of downtime vs. the cost of the DR infrastructure. For Takeda's clinical trial data pipelines, we use Warm Standby because a 4-hour outage could delay trial reporting deadlines worth millions. For internal tooling, Backup & Restore is sufficient. I always start by asking: what's an hour of downtime worth to the business? If it's $50K/hour, Warm Standby or Active-Active pays for itself. If it's $500, Backup & Restore is the right answer."

### Q4. Contrast Monolithic vs. Microservices. Explain Event-Driven Architecture.  `◈ Deep`

The trap here is being dogmatic. Architects who say "microservices are always better" lose points. Show nuance. Dimension Monolith Microservices Deployment Single deployable unit. Fast for small teams. Independent deployments. Enables CI/CD per service. Scaling Scale everything or nothing. Scale only the bottleneck service. Fault isolation One bug can crash everything. Failure bounded to one service (with proper circuit breakers). Operational cost Low — one app, one log stream. High — service mesh, distributed tracing, N databases. Best for Early-stage products, small teams, well-understood domains. Large teams, diverse scaling needs, mature domain models. Event-Driven Architecture Services communicate via events published to a broker (SNS, SQS, EventBridge, Kafka). Producers don't know consumers — loose coupling. Enables: async processing, fan-out, audit trails, temporal decoupling. "I'd add a third option that's often the right answer: the modular monolith — single deployable, but with strict domain boundaries enforced at the code level. You get monolith simplicity today and an extraction path to microservices when load demands it. At Takeda we run a hybrid: core platform services are microservices on EKS, but some internal tools are deliberately modular monoliths because the team is two people and the operational overhead of 12 services would kill velocity. Event-driven patterns come in where we need async fan-out — e.g., a pipeline completion event on EventBridge triggers S3 archiving, Slack notification, and audit log write in parallel without the pipeline knowing any of those consumers exist."

### Q5. Compare SQL vs. NoSQL from a Solutions Architect perspective.  `◈ Deep`

Don't recite "SQL has tables, NoSQL has documents." Speak to access patterns, consistency models, and scaling limits . Factor SQL (RDS/Aurora) NoSQL (DynamoDB) Data model Normalized, relational. Good for complex joins. Denormalized, key-access. Design around access patterns first. Consistency ACID transactions. Strong consistency. Eventually consistent by default. Strong consistency opt-in per read. Scaling Vertical primary. Read replicas for reads. Aurora up to 128TB, 15 replicas. Horizontal, unlimited. On-Demand mode scales to any throughput in seconds. Latency 1–10ms typical query latency. Single-digit ms at any scale. Consistent. Choose when Complex queries, reporting, relationships, ACID needed. Known access patterns, massive scale, low-latency SLAs, simple key-value or document lookups. "The architect's decision is driven by access pattern analysis, not data shape. If the team will write ad-hoc queries and the schema will evolve, SQL wins — flexibility matters more than performance. If the application has 3 known access patterns and needs to scale from day one, DynamoDB wins — you pre-bake the patterns into the table design with partition and sort keys. The anti-pattern I've seen repeatedly: teams start with DynamoDB because it's 'serverless and scalable,' then discover at 6 months they need a join and they're stuck. Know your query patterns before choosing the database."

### Q6. How would you architect a fully available and fault-tolerant application on AWS?  `⚡ High Freq` `◎ Scenario`

Availability = % of time service is accessible. Fault tolerance = ability to continue operating despite failures. Fault tolerant is a stronger guarantee. 1 Eliminate single points of failure at every layer. Multi-AZ deployment for compute (EKS node groups across 3 AZs), data (Aurora Multi-AZ), and networking (NAT Gateway per AZ, not shared). 2 Design for graceful degradation. Circuit breakers (AWS App Mesh / Istio) prevent cascade failures. Read from cache (ElastiCache) when DB is degraded. Return stale data rather than 500 errors. 3 Decouple with async patterns. SQS queues absorb traffic spikes and decouple producers from consumers. A downstream service outage doesn't propagate upstream. 4 Immutable infrastructure + rapid replacement. EC2 instances are cattle, not pets. ASGs replace unhealthy nodes automatically. EKS self-heals pods. Recovery time = seconds, not hours. 5 Health checks everywhere. ALB target health checks, EKS liveness/readiness probes, Route 53 health checks for cross-region DNS failover. 6 Chaos engineering in staging. AWS Fault Injection Simulator. Validate that the theory matches reality before production. "The difference between availability and fault tolerance becomes real under fire: an available system recovers from failure. A fault-tolerant system never stops serving — it handles the failure in-flight. For critical workloads at Takeda, I target fault tolerance for the read path (always return something) and high availability for the write path (recover within 30 seconds). That asymmetry dramatically simplifies the architecture while meeting SLAs."

### Q8. What is your favorite AWS service? How will you improve it?  `★ STAR`

This is a disguised depth test. Pick IAM OIDC Federation — it shows you care about security architecture, not just convenience. "My favorite is IAM OIDC Federation — specifically the integration between GitHub Actions and AWS via OIDC identity providers. Here's why: it eliminated an entire class of security risk we had with long-lived IAM access keys in CI/CD pipelines. With OIDC, GitHub Actions gets a short-lived token scoped to the specific repo and branch, exchanges it for temporary AWS credentials, and the token expires after the job. Zero static secrets to rotate, leak, or audit. But here's where I'd improve it: the trust policy syntax for OIDC is brittle and error-prone. You have to manually write StringLike conditions for repo and branch matching. A mis-scoped trust policy can grant any GitHub repo in the org access to your AWS account. I'd love to see AWS provide a structured IAM console wizard for OIDC federation with explicit repo/branch/environment scoping, and a policy linter that flags overly broad OIDC trust conditions the same way Access Analyzer flags public S3 buckets. The security benefit is enormous but the developer experience is rough enough that teams avoid it — and fall back to access keys." "My favorite is S3 because it stores files" — this wastes the question. Pick something that reveals architectural maturity.

### Q11. Your entire production workload runs in a single AWS region — design a multi-region DR strategy with minimal downtime.  `◎ Scenario` `◈ Deep`

This is a design-under-constraints question. Show you understand the phased migration approach rather than jumping to Active-Active. 1 Define RTO/RPO first. The answer changes completely based on whether the business can tolerate 1 hour vs. 5 minutes of data loss. Get these numbers before designing. 2 Start with Pilot Light (phase 1). Enable Aurora Global Database cross-region replication — <1s lag. Deploy minimal compute in DR region via existing Terraform. Cost: ~15% of primary. 3 Add Route 53 health checks. Configure failover routing policy: primary = us-east-1, secondary = us-west-2. Route 53 checks ALB health every 30s. Automatic DNS failover within 60–90s of primary failure. 4 Automate the failover runbook. Lambda function triggered by Route 53 health alarm: promotes Aurora global secondary to primary, scales up DR EKS node groups, updates any region-specific configs. Target: <10 minute RTO automated. 5 Test quarterly via GameDay. Actual DNS failover to DR region with synthetic traffic. Measure actual RTO vs. target. "The trap is building Active-Active before you need it — it's expensive and operationally complex. I'd implement Pilot Light as the first milestone: Aurora Global Tables for near-zero RPO, Route 53 health-check failover for automated RTO under 10 minutes. That's 80% of the benefit at 20% of the cost. If after 6 months the business validates it needs RTO under 60 seconds, then we invest in Warm Standby or Active-Active. The other thing I'd insist on: the runbook must be code, not a document. A Lambda-based automated failover tested monthly beats a 40-page PDF that no one has read."

### Q2. How to design for both regions active at once with separate access control?  `◈ Deep` `◎ Scenario`

This is Active-Active with data sovereignty or access control separation — common in regulated industries or global products. 1 Traffic routing: Route 53 Geolocation or Latency routing — EU users → eu-west-1, US users → us-east-1. Weighted routing for canary across regions. 2 Data layer: DynamoDB Global Tables or Aurora Global Database for active-active data replication. Conflict resolution strategy: "last writer wins" for DynamoDB, application-level for Aurora. 3 Access control separation: Separate AWS accounts per region under AWS Organizations. IAM roles scoped per account — EU team cannot assume roles in US account. SCPs at OU level enforce data residency (e.g., deny actions that replicate data outside EU). 4 KMS: Region-specific KMS keys — data encrypted in eu-west-1 cannot be decrypted by keys in us-east-1 unless explicitly granted. Strong data sovereignty control. "For GDPR compliance specifically, I'd add an SCP: Deny s3:ReplicateObject if destination bucket is outside eu-west-1. This is a hard guardrail that survives IAM misconfiguration — even an admin can't accidentally replicate EU personal data to a US region. The SCP sits above IAM in the policy evaluation hierarchy, so it's enforced regardless of what the application or developer does."

### Q5. What does 3/3 status check mean on an EC2 instance?  `⚡ High Freq`

Simple question with a depth trap — most people know 2/2. The third check is impairment status added in 2023. Check What it verifies Who fixes it System Status Check Underlying AWS hardware/hypervisor. Network connectivity to instance, power, software issues on the host. AWS. Stop+start migrates to healthy host. Instance Status Check OS-level: kernel, networking config, file system, application. Instance is reachable and responding. You. Reboot, check OS logs, fix config. ### Q6. Share AMI with child accounts? Shared AMI instance launches and terminates? Diagnose shared AMI issues?  `◎ Scenario`

Three connected questions — answer them as a single flow showing you understand the AMI + snapshot permission model . Sharing an AMI aws ec2 modify-image-attribute \ --image-id ami-0abc123 \ --launch-permission "Add=[{UserId=CHILD_ACCOUNT_ID}]" # CRITICAL: Also share the underlying EBS snapshot aws ec2 modify-snapshot-attribute \ --snapshot-id snap-0abc123 \ --attribute createVolumePermission \ --operation-type add \ --user-ids CHILD_ACCOUNT_ID Why shared AMI instance launches then terminates Missing snapshot permission — AMI shared but underlying EBS snapshot not shared. Instance fails to attach root volume → immediate termination. Most common cause. AMI uses encrypted snapshot with KMS key not shared to child account. Decrypt fails silently → terminate. Instance profile in child account lacks permissions required by user-data script at boot. Target instance type not available in child account's region (e.g., newer family not enabled). Diagnosis steps Check EC2 System Log: aws ec2 get-console-output --instance-id i-xxx Verify snapshot is shared: aws ec2 describe-snapshot-attribute --snapshot-id snap-xxx --attribute createVolumePermission Check KMS key policy for cross-account grants if AMI is encrypted. Review CloudTrail in child account for RunInstances → look for error codes. "The instant-terminate scenario is 9 times out of 10 a missing snapshot permission. The AMI is visible to the child account but the actual EBS snapshot backing it isn't shared — so when EC2 tries to create the root volume, it gets an access denied, and the instance terminates. The fix is two operations: share the AMI and share each snapshot in the AMI's block device mapping. For encrypted AMIs, you also need to grant the child account's service role kms:Decrypt and kms:CreateGrant on the KMS key. I've encoded this as a Terraform module that handles all three permissions atomically — you can't share the AMI without also correctly configuring the snapshot and KMS permissions."

### Q19. Users report intermittent failures across microservices, but logs from individual services look normal. How do you identify root cause?  `◎ Scenario` `◈ Deep`

This is a distributed systems debugging question. Individual service logs looking normal is the key clue — the failure is in the interactions, not the services themselves. 1 Distributed tracing first. Individual logs can't see cross-service latency. Open a Datadog APM or AWS X-Ray trace for a failing request. The trace shows end-to-end span latency — the broken link will be visible as a red span or unusual latency spike between two services. 2 Look for patterns in the intermittency. What % of requests fail? Is it correlated with time of day (load-related)? Specific services or endpoints? Specific AZ? Specific node? Correlating failure dimensions narrows the search dramatically. 3 Network-level investigation. VPC Flow Logs for packet drops. Check if retried requests succeed (TCP retransmits → network issue vs. application logic). ALB access logs for 504s upstream. 4 Infrastructure-level checks. Node memory pressure causing OOM kills that restart pods (pod restarts don't show in service logs — check Kubernetes events). Noisy neighbor on the same node. DNS resolution latency (CoreDNS issues in EKS are a classic intermittent failure cause). 5 Dependency health. External dependencies (RDS, ElastiCache, third-party APIs) may be intermittently slow without erroring. Instrument all external calls with latency metrics — p99 spikes that don't cross error thresholds still cause timeouts upstream. "The phrase 'logs look normal' tells me the problem is likely infrastructure or network, not application code. My first move is always distributed tracing — it collapses a multi-service investigation into a single waterfall diagram. The second most common cause I've seen for intermittent microservice failures that don't show in individual logs: CoreDNS rate limiting in EKS. Kubernetes service discovery relies on DNS, and if CoreDNS is overwhelmed, service-to-service calls time out silently — the calling service sees a timeout, not a DNS error. Mitigation: NodeLocal DNSCache as a DaemonSet, which puts a DNS cache on every node and reduces CoreDNS load by 80–90%."

### Q1. How would you make a subnet public or private?  `⚡ High Freq`

The distinction is purely about route table configuration , not the subnet itself. Type Route table requirement Internet path Public subnet Route 0.0.0.0/0 → Internet Gateway (IGW) Direct inbound + outbound. Instance needs a public or Elastic IP. Private subnet Route 0.0.0.0/0 → NAT Gateway (in a public subnet) Outbound only. No inbound from internet. Instance has only private IP. Isolated subnet No 0.0.0.0/0 route at all None. Only reachable within VPC. Use for databases, internal services. "A subnet is 'public' purely because of its route table — specifically a 0.0.0.0/0 route pointing to an IGW. The subnet itself has no public/private property. The three-tier model I use at Takeda: public subnets for ALBs and NAT Gateways only — no application servers ever get public IPs. Application tier in private subnets with NAT Gateway for outbound-only internet. Database tier in isolated subnets with no internet route at all — the only way in is from the app subnet via security group rules. Defense-in-depth through subnet isolation costs nothing but adds a meaningful layer."

### Q2. Internet Gateway is attached to a subnet or a VPC?  `⚡ High Freq`

Classic trick question. The answer is VPC , not subnet. "The IGW is attached to the VPC — one IGW per VPC, and it's horizontally scaled and redundant by AWS. The confusion comes from thinking public subnets 'have' the IGW, but what they actually have is a route table entry: destination 0.0.0.0/0, target igw-xxxxxxxx. Multiple subnets in the same VPC can share that route to the same IGW. The subnet becomes 'public' by associating a route table that includes that IGW route — the IGW itself never moves. Understanding this distinction matters when you're troubleshooting why an EC2 in what looks like a public subnet can't reach the internet: check the route table association on the subnet, not the IGW."

### Q3. Where are the outgoing rules for a private subnet needing internet access?  `⚡ High Freq`

Tests whether you understand the NAT Gateway placement model and routing asymmetry. "Two route tables work together: the private subnet's route table has 0.0.0.0/0 → NAT Gateway. The NAT Gateway itself lives in a public subnet whose route table has 0.0.0.0/0 → IGW. So outbound internet traffic from a private instance flows: instance → NAT GW (source NAT applied, private IP replaced with NAT GW's Elastic IP) → IGW → internet. Return traffic reverses through the NAT GW which maintains connection state. The NACL rules at the private subnet level also need to permit outbound on the required port — NACL is stateless so you also need inbound allow for the ephemeral port range (1024–65535) for return traffic. Security Groups, being stateful, handle the return automatically." Forgetting that NACLs are stateless — teams open outbound 443 but forget the inbound ephemeral port range, then debug for hours.

### Q5. Why do you need Transit Gateway when we have VPC Peering? How is TGW beneficial?  `⚡ High Freq` `◈ Deep`

This tests your understanding of N² peering problem and operational scalability. VPC Peering 1:1 connection between 2 VPCs. Not transitive — if A peers with B and B peers with C, A cannot reach C. N VPCs = N*(N-1)/2 peering connections. 10 VPCs = 45 connections. Transit Gateway Hub-and-spoke model. N VPCs connect to 1 TGW = N connections total. Transitive routing. Supports VPN, DX, peering between TGWs across regions/accounts. When VPC Peering is still better 2–3 VPCs only, simple routing — VPC peering is free (no hourly charge), TGW costs ~$0.05/hour per attachment + data processing fee. Cross-account peering where you want explicit, minimal blast radius connectivity. When you need lower latency — TGW adds a small routing hop. "VPC Peering breaks at scale for two reasons: the N² connection problem and non-transitivity. At Takeda we have 15+ VPCs — managing 100+ peering connections, each with its own route table entries and security group rules, is operationally impossible. Transit Gateway collapses that to 15 attachments and a single routing domain. Beyond that, TGW adds capabilities peering never had: multicast support, centralized egress to a shared NAT Gateway, inter-region peering (TGW peering across regions), and a single place to attach Direct Connect via a Direct Connect Gateway. The cost is real — TGW has hourly + data charges — but at 10+ VPCs the operational savings far outweigh it."

### Q8. Difference between Security Group vs NACL — and the follow-up 80% miss  `⚡ High Freq`

### Q12. How does AWS IAM evaluate policies? What is policy evaluation logic order?  `◈ Deep` `⚡ High Freq`

One of the most important security questions. The evaluation order has a strict priority hierarchy and explicit deny always wins. 1 Explicit DENY anywhere = DENY. Full stop. No other policy can override an explicit deny. Check SCP, resource policy, identity policy, permission boundary — any explicit deny ends evaluation immediately. 2 Service Control Policy (SCP) — Must have an Allow in SCP. SCPs don't grant permissions — they set the maximum permissions boundary for an account. Even if all other policies allow, an SCP that doesn't allow the action = DENY. 3 Resource-based policy — If the resource (S3 bucket, KMS key, SQS queue) has a resource policy that allows the principal, access is granted even without an identity policy (for same-account). Cross-account requires BOTH resource policy AND identity policy to allow. 4 Permission boundary — Sets the maximum permissions for an IAM entity. The effective permission is the intersection of identity policies AND permission boundary. 5 Identity-based policy — Inline or managed policies attached to the user/role. Must have an explicit Allow here. 6 Session policy — For assumed roles, the session policy (passed at assume-role time) further restricts permissions. Cannot grant beyond the role's identity policies. "The mantra is: default deny, explicit deny wins, SCPs set the ceiling, permission boundaries set the per-identity ceiling, then identity policies grant within all those constraints. The cross-account case is the most common source of bugs: a developer adds an S3 bucket policy allowing Account B, but the IAM role in Account B doesn't have s3:GetObject in its identity policy. Access denied — both sides must allow for cross-account. Same-account is different: a resource policy allowing a same-account principal is sufficient without an identity policy, because the resource policy implicitly trusts the account."

### Q13. Difference between identity-based and resource-based policies?  `◈ Deep`

Type Attached to Specifies Examples Identity-based IAM user, group, role What the principal CAN do Managed policies, inline policies, permission boundaries Resource-based The AWS resource itself Who CAN access THIS resource and what they can do S3 bucket policy, KMS key policy, SQS queue policy, Lambda resource policy "The critical difference is the Principal field. Identity-based policies have no Principal — they're attached to a principal and implicitly apply to it. Resource-based policies require an explicit Principal: which accounts, roles, or services can access this resource. This is what enables cross-account access without VPC peering: you grant another account's role in the resource policy, and they grant the action in their identity policy. Resource-based policies are also what allow AWS services to call your resources — Lambda needs a resource policy to be invoked by API Gateway; SQS needs a resource policy for SNS to send messages to it."

### Q14. How does AWS STS work internally? How do you implement cross-account access securely?  `◈ Deep`

STS is the temporary credential vending service — everything in AWS that needs short-lived credentials goes through STS. How STS works You call sts:AssumeRole with the target role ARN and an optional external ID and session policy. STS verifies your identity, checks the role's trust policy to confirm you're an allowed principal, and validates SCPs/permission boundaries. Returns a temporary set of credentials: AccessKeyId, SecretAccessKey, SessionToken — valid for 15 min to 12 hours. The temporary credentials are scoped to the assumed role's permissions, further limited by any session policy passed at assume time. Cross-account access securely # Account A (source) has a role with this trust policy: { "Principal": { "AWS": "arn:aws:iam::ACCOUNT-B:role/DeployRole" }, "Action": "sts:AssumeRole", "Condition": { "StringEquals": { "sts:ExternalId": "unique-secret-per-customer" } } } "The ExternalId condition is critical for third-party cross-account access — it prevents the confused deputy problem where a third-party SaaS provider could be tricked into assuming your role on behalf of a malicious customer. For internal cross-account (your own accounts in an Org), I use the aws:PrincipalOrgID condition instead — it restricts assume-role to principals within your AWS Organization, blocking external entities even if they somehow obtain your role ARN. Combined with SCPs that deny AssumeRole from outside the org, this creates defense-in-depth for cross-account trust."

### Q15. What is AWS Organizations and SCP? How do Service Control Policies work?  `⚡ High Freq` `◈ Deep`

SCPs are the guardrails that sit above IAM — they constrain what even Account Admins can do. SCPs apply to all IAM principals in member accounts (including root user, except for management account). SCPs do NOT grant permissions — they define the maximum allowed permissions. An SCP Allow doesn't mean access is granted; an SCP Deny always blocks. Inheritance: SCPs at the root apply to all accounts. OUs inherit from parent OU. Effective SCP = intersection of all SCPs in the path from root to account. Real SCP examples # Deny all actions outside approved regions { "Effect": "Deny", "Action": "*", "Resource": "*", "Condition": { "StringNotEquals": { "aws:RequestedRegion": ["us-east-1", "eu-west-1"] } } } # Deny disabling CloudTrail { "Effect": "Deny", "Action": ["cloudtrail:StopLogging", "cloudtrail:DeleteTrail"], "Resource": "*" } "My SCP strategy at Takeda uses three layers: region restriction (deny all actions outside approved regions — data residency compliance), immutable audit (deny CloudTrail stop/delete, Config recorder disable — ensures audit trail can't be tampered with), and data exfiltration prevention (deny S3 replication to buckets outside the Org, deny creating internet-facing resources in production accounts). These SCPs are tested via a CI pipeline that uses AWS IAM Policy Simulator — any SCP change must pass a test suite that validates both intended blocks and unintended allows before merging."

### Q16. How do you enforce security guardrails in AWS?  `★ STAR`

Defense-in-depth answer: multiple layers at different points in the lifecycle. 1 Preventive — SCPs: Hard stops at the Org level. No one in the account can violate these, including admins. 2 Preventive — IaC policy as code: Checkov / tfsec / OPA in CI pipeline. Terraform plan fails if security policy violated before any resource is created. 3 Preventive — Kubernetes admission: OPA Gatekeeper / Kyverno policies enforce pod security standards, required labels, IRSA requirements, image registry allowlisting. 4 Detective — AWS Config Rules: Continuous compliance evaluation. High severity = immediate remediation, medium = alert + ticket. "The key principle: push left. A guardrail that catches violations at the Terraform plan stage costs zero to remediate. A guardrail that catches it in production costs an incident. I measure the effectiveness of each guardrail layer: how many violations are caught at IaC vs. Config vs. runtime? If too many reach Config, the IaC layer needs tightening. The ideal state: Config and GuardDuty rarely fire because SCPs and IaC policies caught everything earlier."

### Q17. How do you secure S3 buckets at scale? How do you prevent data exfiltration from S3?  `⚡ High Freq`

### Q18. What is AWS Shield vs WAF? How does AWS WAF protect applications?  `⚡ High Freq`

### Q26. How do you facilitate cross-account connectivity? What configuration is required?  `◈ Deep`

"Three patterns in order of preference: (1) PrivateLink — expose a specific service endpoint across accounts. Consumer creates an Interface Endpoint; provider creates an NLB-backed endpoint service. No network peering required, no CIDR conflicts possible. Best for service-to-service connectivity. (2) Transit Gateway cross-account attachment — share TGW resource via RAM (Resource Access Manager), attach VPCs from different accounts. All accounts can reach all others through the TGW route table. Best for full network connectivity. (3) VPC Peering — direct 1:1 connection. Requester initiates, accepter accepts. Account A: Create peering request to Account B VPC. Account B: Accept the peering request. Both accounts: Add route table entries — destination = peer VPC CIDR, target = peering connection ID. Both accounts: Update security groups to allow traffic from peer VPC CIDR. DNS resolution: Enable DNS resolution for VPC peering if private hosted zones need to resolve across accounts.

### Q3. Docker best practices for image creation. How to reduce Docker image size?  `⚡ High Freq`

### Q6. How do you update EKS clusters? How to ensure zero downtime during EKS upgrade?  `⚡ High Freq` `◎ Scenario`

EKS upgrade order 1 Read EKS release notes — API deprecations, breaking changes. Kubernetes releases deprecate APIs each minor version. Identify deprecated API objects in your manifests using pluto or kubent . 2 Update control plane first — aws eks update-cluster-version . One minor version at a time (1.27 → 1.28, not 1.27 → 1.30). Nodes are replaced via rolling drain. Zero downtime guarantees PodDisruptionBudget (PDB): minAvailable: 1 on all deployments. Node drain respects PDBs — won't evict the last pod of a deployment. Multiple replicas: At least 2 replicas per deployment across multiple AZs. Topology spread constraints to ensure AZ distribution. Readiness probes: New pods on new nodes only receive traffic after passing readiness probe. Old pods removed from LB target group before termination. terminationGracePeriodSeconds: Long enough for in-flight requests to complete (typically 30–60s). Pre-stop hook to drain connections. "The one thing most teams miss: PDBs. Without PDBs, a node drain during upgrade can evict all pods of a deployment simultaneously if they happen to be on the same node. ### Q9. K8s troubleshooting scenarios — pod in CrashLoopBackOff  `◎ Scenario` `⚡ High Freq`

1 kubectl describe pod <name> -n <ns> — check Events section. "Back-off restarting failed container." Look at Last State exit code — 137 = OOMKilled, 1 = app crash, 139 = segfault. 2 kubectl logs <pod> --previous — logs from the crashed container instance. This is the key command — without --previous you see the new (possibly not-yet-crashed) container. 3 Check resource limits — OOMKilled (exit 137) = memory limit too low or memory leak. Increase memory limit or investigate heap dump. 4 Check liveness probe — if the probe fails too early during startup, Kubernetes kills and restarts the pod. Set initialDelaySeconds longer than app startup time, or use a startup probe. 5 Check secrets/configmaps — missing env var or secret causes app to exit immediately. kubectl get events --sort-by=.lastTimestamp shows mount failures. 6 kubectl debug -it <pod> --image=busybox --copy-to=debug-pod — create a copy of the failing pod with a different entrypoint to investigate interactively. "Exit code is the first diagnostic. 137 = OOMKilled (always). 1 = application crashed — go to logs. 0 in CrashLoopBackOff = the container exited cleanly but shouldn't — usually a wrong CMD or ENTRYPOINT. The most common cause I see at Takeda: a secret referenced in env vars doesn't exist in the namespace. The pod starts, tries to mount the secret, fails, crashes. kubectl describe pod Events section says 'secret not found' clearly. The fix is either creating the missing secret or using External Secrets Operator to sync from AWS Secrets Manager automatically."

### Q12. Experience with probes? How are they different? Why are probes needed?  `⚡ High Freq`

Probe What it checks Failure action When to use Liveness Is the container still alive and not deadlocked? Kill and restart the container Apps that can enter an unrecoverable state (deadlock, corrupted state) but the process is still running. Readiness Is the container ready to serve traffic? Remove from Service endpoints (no traffic) Always. Ensures traffic only goes to warmed-up, healthy pods. Critical during deployments. Startup Has the container finished initial startup? Kill and restart if not successful within timeout Slow-starting apps (JVM, ML model loading). Replaces liveness during startup to avoid false kills. "The startup probe is the most underused of the three. Without it, a JVM app that takes 90 seconds to start will be killed by liveness probe (which fires every 10s after initialDelaySeconds=30). The fix is always 'increase initialDelaySeconds,' which is a blunt instrument — it delays failure detection for genuinely crashed pods. Startup probe is elegant: it gives the container a generous startup window (successThreshold=1, failureThreshold=30, periodSeconds=10 = 300s to start), then hands off to liveness probe which can be aggressive (10s intervals). Fast detection of crashes after startup, patient during startup."

### Q13. What is HPA (Horizontal Pod Autoscaler)? Difference between HPA, KEDA, and Karpenter? Why still need HPA if Karpenter is used?  `◈ Deep` `⚡ High Freq`

Tool Scales what Metric source Use case HPA Pod replicas CPU/Memory (Metrics Server), custom metrics (Prometheus adapter) Workloads that scale on utilization. Simple and battle-tested. KEDA Pod replicas (including scale to 0) External event sources: SQS depth, Kafka lag, Datadog metrics, cron Event-driven workloads. Scale workers based on queue depth, not CPU. ### Q14. How does Service A communicate with Service B inside Kubernetes?  `◈ Deep`

# Service B exposed via ClusterIP Service # Service A calls: http://service-b.namespace-b.svc.cluster.local:8080 # DNS resolution path: Pod A → CoreDNS (kube-dns) → resolves service-b.namespace-b.svc.cluster.local → returns ClusterIP (e.g., 172.20.0.45) → kube-proxy (iptables/IPVS rules) → selects healthy pod endpoint → traffic hits Service B pod "The full path: Service A's pod DNS resolver queries CoreDNS for the service FQDN. CoreDNS returns the ClusterIP — a virtual IP that exists only in iptables rules on every node (managed by kube-proxy). When traffic hits the ClusterIP, iptables rules perform DNAT — destination NAT rewrites the destination IP to one of the actual pod IPs from the Endpoints object. For same-namespace calls, you can omit the namespace: http://service-b:8080 . For cross-namespace, use the full FQDN. The subtle thing worth mentioning: kube-proxy in iptables mode has an O(n) rule lookup problem at scale — 10,000 services = 10,000 iptables rules traversed per connection. IPVS mode (hash table O(1)) or Cilium's eBPF data plane solves this for large clusters. At Takeda with 200+ services, we switched to Cilium and saw meaningful latency improvements on service-to-service calls."

### Q15. How does a pod securely connect to S3 without hardcoding credentials?  `⚡ High Freq` `◈ Deep`

I use IRSA for every pod that needs AWS access at Takeda. The IAM role is created in the same Terraform module as the Kubernetes ServiceAccount — they're deployed atomically. ### Q17. What are taints and tolerations? What are dedicated nodes?  `◈ Deep`

"Taints repel pods from nodes; tolerations allow pods to schedule on tainted nodes. Together they implement dedicated node pools. Example: GPU nodes get a taint gpu=true:NoSchedule — no pod can schedule on them unless it has a matching toleration. Only ML workload pods have the toleration. This prevents general-purpose pods from landing on expensive GPU nodes and wasting GPU capacity. Dedicated nodes pattern: Karpenter NodePool with specific instance types + taints for the workload type. Three use cases at Takeda: GPU nodes for ML jobs (taint: gpu), high-memory nodes for data processing (taint: highmem), and system nodes for platform components like monitoring (taint: platform=true). Combined with node affinity (nodeSelector or requiredDuringSchedulingIgnoredDuringExecution), workloads are guaranteed to land on the right node type. The difference between taints and node affinity: taints push pods away, affinity pulls pods toward specific nodes — you often need both."

### Q18. If I want to run two applications in the same EKS cluster but keep them isolated — how would you design it?  `◎ Scenario` `◈ Deep`

Multi-tenancy within a Kubernetes cluster — multiple isolation layers needed. 1 Namespace isolation: Separate namespaces per team/application. Kubernetes RBAC scoped to namespace — Team A cannot see or modify Team B's resources. 2 Network policies: Default-deny ingress + egress in each namespace. Explicit allow rules for only required cross-namespace communication. Calico or Cilium enforces at the kernel level. 3 Resource quotas: ResourceQuota per namespace — limits CPU, memory, PVC count, pod count. Prevents one team from starving the other. 4 Node isolation (strong isolation): Dedicated node pools with taints per team. Team A pods can only schedule on Team A nodes. Physical separation of compute. 5 OPA Gatekeeper policies: Enforce that pods in namespace team-a must use images from team-a's ECR repo only. Prevents cross-namespace image sharing that could introduce supply chain risk. "The isolation level you need depends on the threat model. For teams from the same org with shared trust: namespace + RBAC + network policies + resource quotas is sufficient. For regulatory isolation (e.g., PCI workload and non-PCI workload in same cluster): you need dedicated nodes with taints to achieve compute-level isolation — namespaces share the same kernel. For truly adversarial tenants (SaaS multi-tenancy): separate clusters. At Takeda, clinical GxP workloads run in a dedicated namespace with dedicated nodes and strict network policies — non-GxP workloads cannot reach them by any network path, enforced at the Cilium eBPF level."

### Q20. What would you do if Amazon EKS runs out of IP addresses?  `◎ Scenario` `◈ Deep`

EKS with VPC CNI assigns each pod a real VPC IP from the node's subnet. IP exhaustion is a real operational risk in EKS. Why it happens VPC CNI pre-allocates IPs to each node (warm IP pool). A node with 8 pods might hold 12 reserved IPs. IP consumption is higher than pod count suggests. Subnets created with small CIDRs (/24 = 251 usable IPs, minus 5 reserved = 246) fill up with large node counts. Solutions 1 Immediate: Enable VPC CNI prefix delegation — each node gets a /28 prefix (16 IPs) per ENI instead of individual IPs. Increases pod density per node significantly without new subnets. 2 Add secondary CIDR: Associate a secondary CIDR block (e.g., 100.64.0.0/16 — RFC 6598 range) to the VPC. Create new /24 subnets from this range. Assign these as node subnets. Pods get IPs from the secondary range, freeing primary CIDR for other resources. 3 Custom networking: VPC CNI custom networking mode — pods get IPs from a secondary subnet different from the node's primary subnet. Decouples pod IP space from node IP space. 4 Alternative CNI: Cilium in overlay mode — pods get IPs from an internal CIDR not tied to VPC. No VPC IP consumption per pod. Requires replacing VPC CNI entirely. "The fastest fix with no architecture change: enable prefix delegation on the VPC CNI. One command: kubectl set env daemonset aws-node -n kube-system ENABLE_PREFIX_DELEGATION=true . This immediately increases available IPs per node by using /28 prefixes. For long-term: always create EKS node subnets at /22 or larger and use the secondary CIDR (100.64.0.0/16) strategy from day one. I've built this into Takeda's EKS Terraform module as a default — new clusters provision /22 node subnets and enable prefix delegation by default, so we never hit this problem again."

### Q22. If you have 4 microservices, would you deploy them in 4 separate pods or a single pod? Why?  `⚡ High Freq`

"4 separate pods — one container per pod is the Kubernetes design intent and best practice for microservices. Each pod is the unit of scheduling, scaling, and failure isolation. 4 microservices in one pod means: if one crashes, all restart. If one needs 3x scale, all scale. ### Q24. How does Rollback Deployment work internally?  `◈ Deep`

"Kubernetes deployments maintain a revision history of ReplicaSets. When you deploy a new version, a new ReplicaSet is created and scaled up while the old one is scaled down (rolling update). The old ReplicaSet is kept at 0 replicas — this is your rollback target. kubectl rollout undo deployment/myapp simply swaps — scales the previous ReplicaSet back up and scales the current one down. The rollback history depth is controlled by revisionHistoryLimit (default 10). kubectl rollout history deployment/myapp shows the available revisions. With ArgoCD, rollback is different — it's Git-driven. Argo treats the Git commit as the source of truth. A rollback is literally reverting the Git commit and letting ArgoCD sync the previous state. This is more auditable: the rollback shows in Git history with who did it and when. With Argo Rollouts, an automated rollback is triggered by the analysis template failure — no human required, just a Prometheus/Datadog query that exceeds threshold."

### Q29. Kubernetes best practices — the 10 that differentiate senior engineers  `⚡ High Freq`

### Q33. How do you ensure zero downtime deployments?  `⚡ High Freq`

### Q43. How do you ensure zero downtime deployments across all layers?  `⚡ High Freq`

Kubernetes layer: PodDisruptionBudgets (minAvailable: 1), readiness probes (traffic only to ready pods), preStop hook with sleep 5 (ALB deregisters pod before SIGTERM), terminationGracePeriodSeconds ≥ longest request duration. ALB layer: Connection draining on target deregistration (default 300s). New pods register with ALB via readiness gate — no traffic until healthy. Database layer: Backwards-compatible schema changes. Never drop a column in the same deploy as removing the code that uses it. Blue-green DB migration: add column → deploy code that reads both → backfill → remove old column in next release. DNS layer: Low TTLs (60s) before planned failovers. Route 53 health checks for automatic failover. Deployment strategy: Canary or blue-green via Argo Rollouts — never route 100% of traffic to new version until metrics validate it. "The most common source of downtime during 'zero-downtime deployments' is the preStop gap: when Kubernetes sends SIGTERM, the pod is removed from the Endpoints object — but ALB still has the IP in its routing table for up to 30 seconds (deregistration delay). Requests land on a pod that's already shutting down → 502 errors. The fix: preStop hook with a sleep matching the ALB deregistration delay (typically 10–30s). This keeps the pod alive long enough for ALB to drain existing connections before the process exits. Without this, every rolling deployment produces a brief 502 spike visible in CloudWatch."

## Scalability & Performance

### Q1. How can you make your application scalable for a big traffic day?  `⚡ High Freq` `◎ Scenario`

This is an end-to-end systems design question — touch every layer. A big traffic day (Black Friday, product launch, Super Bowl ad) is a known event, so both pre-warming and reactive scaling apply. 1 Load test first. Run k6 or Locust at 3–5x expected peak before the event. Find the bottleneck (DB connections, app CPU, cache miss rate) before traffic does. Fix it with data, not guesswork. 2 Pre-scale compute. EKS: manually bump HPA minReplicas 24 hours before. Karpenter will provision nodes. ASG: set MinSize to peak capacity — don't wait for autoscaling lag. Schedule the scale-up via EventBridge cron. 3 CDN and caching. CloudFront pre-warming for known URLs. ElastiCache in front of the DB — cache hit rate >95% means the DB sees 1/20th of the traffic. Ensure TTLs are set correctly before the event. 4 Database layer. Aurora reader endpoints for read traffic. Connection pooling via RDS Proxy — prevents connection exhaustion at high concurrency. Read replicas pre-provisioned, not created on demand (promotion takes minutes). 5 Async offloading. Any non-critical synchronous work (email confirmation, analytics event, audit log) goes to SQS. The web tier returns immediately; workers drain the queue. Queue acts as a shock absorber. 6 Circuit breakers and graceful degradation. If a downstream service (payment processor, recommendation engine) is slow, return a degraded response rather than hanging all threads. A 200 with "recommendations unavailable" beats a 504. 7 War room ready. On-call team briefed, rollback plan documented, feature flags to disable non-critical features under extreme load, and CloudWatch dashboards on the big screen. "The biggest mistake teams make for traffic events: they fix the compute layer but forget the database. At peak load, connection exhaustion on RDS is the most common failure mode — not CPU. 1,000 pods each opening 10 connections = 10,000 DB connections, which overwhelms a db.r6g.2xlarge that handles ~2,000. RDS Proxy pools connections at the proxy layer — all 1,000 pods share 100 real DB connections. Deploy RDS Proxy before the event, test connection behavior under load, and that bottleneck disappears. At Takeda we tested this explicitly with k6 generating 50k concurrent users against staging — found the connection limit before it could fail in production."

### Q5. What is a Terraform module? Difference between data block and resource block?  `⚡ High Freq`

"A module is a reusable, parameterized Terraform configuration — a function for infrastructure. It encapsulates complexity: our EKS module creates the cluster, node groups, add-ons, IRSA roles, and security groups in one call. Consumers pass variables; the module handles the implementation. Resource block vs data block: resource creates and manages a piece of infrastructure — Terraform owns its lifecycle (create/update/destroy). Data block reads existing infrastructure that Terraform didn't create — it's read-only. Example: data.aws_vpc.existing fetches the VPC ID of a VPC created by another team. resource aws_subnet new creates a subnet in that VPC. The data source is a reference; the resource is an owned object. Using data sources for cross-team dependencies prevents Terraform state coupling between teams."

### Q11. Is a Webhook inbound or outbound, and how does it work in CI/CD?  `⚡ High Freq`

"From the CI/CD system's perspective, a webhook is inbound — GitHub pushes an HTTP POST to your CI system's endpoint when an event occurs (push, PR, merge). The CI system doesn't poll GitHub; GitHub calls it. This is the push model vs. polling model. In GitHub Actions, webhooks are managed internally by GitHub — you define event triggers in the workflow YAML ( on: push: branches: [main] ) and GitHub's infrastructure fires the job when the event matches. For Jenkins, you'd configure the GitHub webhook to POST to https://jenkins.example.com/github-webhook/ and install the GitHub plugin to process the payload. The security consideration: webhook endpoints must validate the payload signature (GitHub signs with a secret using HMAC-SHA256) to prevent anyone from triggering builds by POSTing to your endpoint."

### Q2. Explain the booting process of Linux.  `◈ Deep`

BIOS/UEFI → POST (hardware check) ↓ MBR/GPT (Master Boot Record reads boot sector) ↓ GRUB2 (bootloader — loads kernel + initramfs from /boot) ↓ Kernel initialization - mounts initramfs (temporary root fs in memory) - loads drivers for storage, network - mounts real root filesystem (/) ↓ systemd (PID 1) — replaces old init/SysV - reads /etc/systemd/system/ unit files - parallel service startup (vs sequential in SysV) - targets: multi-user.target, graphical.target ↓ Login prompt (getty) / SSH daemon ready "The critical operational implication: systemd is PID 1, so everything is a child of systemd. systemctl status service shows the service's state, last 10 log lines, and PID. journalctl -u nginx --since today shows all logs for nginx since midnight. systemctl enable service makes it start on boot (creates a symlink in the appropriate target directory). For EC2, this matters when debugging AMI boot issues: the cloud-init service (also managed by systemd) runs user-data scripts. If user-data fails, check journalctl -u cloud-init and /var/log/cloud-init-output.log ."

### Q5. EC2 CPU suddenly spikes — how to debug like a senior engineer  `◎ Scenario` `⚡ High Freq`

### Q7. Docker container works locally but fails in production — why?  `◎ Scenario`

### Q14. Steps to investigate and remediate an issue (production incident process)  `◎ Scenario` `★ STAR`

### Q15. How ensure devs don't overwrite each other's work? Info on enterprise workspaces?  `⚡ High Freq`

### Q22. Any connectivity failures you resolved? Long-standing issues?  `★ STAR`

" Situation: EKS pods intermittently failing to connect to on-prem Oracle DB via Direct Connect, 2-3 times per week, 5-10 minutes each. No correlation with deployments or traffic patterns. Task: Diagnose and permanently fix without disrupting daily data pipelines. Action: Enabled VPC Flow Logs and queried with Athena. Found TCP RST packets from the on-prem firewall at regular intervals. Root cause: on-prem firewall had a 350-second idle TCP timeout; our connection pool kept connections open indefinitely. The firewall silently killed idle connections; the app tried to reuse dead connections and got RST. Fix: configured TCP keepalive on EKS nodes (tcp_keepalive_time=300, shorter than the firewall timeout) and connection pool validation (testOnBorrow=true). Result: Zero connectivity failures in the following 3 months. Added CloudWatch alarm on TCP RST rate as early warning for future firewall drops."

### Q25. chmod 555 — what does it mean? Full permissions reference.  `⚡ High Freq`

chmod 555 # r-xr-xr-x — read+execute for all, NO write for anyone chmod 644 # rw-r--r-- — standard file (owner rw, others r) chmod 755 # rwxr-xr-x — standard executable (owner full, others r+x) chmod 700 # rwx------ — owner only, completely private chmod 400 # r-------- — read-only owner (SSH private key) chmod 777 # rwxrwxrwx — everyone full (DANGEROUS) # Permission math: r=4, w=2, x=1 # 7 = 4+2+1 = rwx, 5 = 4+0+1 = r-x, 4 = 4+0+0 = r-- # Symbolic form chmod u+x file # add execute for owner chmod go-w file # remove write for group and others "chmod 555 = read+execute for owner, group, and others, but no write for anyone including the owner. Use case: a shared utility script that should be executable by all but immutable — even the owner can't accidentally modify it. The permission I enforce on SSH private keys: 400. SSH refuses to use private keys readable by group or others — it's a security check, not a courtesy. In containers: the non-root USER in Dockerfile combined with readOnlyRootFilesystem:true in Kubernetes follows the same principle — minimal write access reduces attack surface."

### Q27. 3-tier architecture steps. Soft vs hard link. Open firewall.  `⚡ High Freq`

## Soft vs hard link ln -s /original /link # soft (symlink) — separate inode, breaks if original deleted ln /original /link # hard link — same inode, survives original deletion ## Open firewall # firewalld (RHEL/Amazon Linux) firewall-cmd --add-port=443/tcp --permanent firewall-cmd --reload # iptables iptables -A INPUT -p tcp --dport 443 -j ACCEPT iptables-save > /etc/iptables/rules.v4 # persist # Security Groups (AWS) — preferred over host-level firewall on EC2 # Add inbound rule: TCP 443, source 0.0.0.0/0 (or specific CIDR) 3-tier architecture on Linux Tier 1: Nginx reverse proxy (public subnet, ports 80/443). Tier 2: Application server — Node.js/Python/Java (private subnet, app port only). Tier 3: Database — PostgreSQL/MySQL (isolated subnet, DB port from app tier only). Each tier communicates only with adjacent tiers via firewall/security group rules. Cloud equivalent: ALB → EKS → RDS, each in progressively isolated subnets. "Hard links are the same file with multiple directory entries pointing to the same inode — deleting the original doesn't affect the hard link because the inode stays referenced. Soft links (symlinks) are pointers to a path — if the target path is deleted or moved, the symlink breaks. Practical use: /etc/nginx/sites-enabled/ contains symlinks to /etc/nginx/sites-available/ — enabling a site creates a symlink, disabling removes it, the config file is never duplicated."
