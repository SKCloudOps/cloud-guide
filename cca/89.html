01 Error Handling & Tool Design Q1, Q7, Q10 ▶
Q1 · Q7 · Uniform vs. structured errors
Always return structured, actionable error metadata D / A
Generic {"isError": true, "text": "Operation failed"} forces the model to guess recovery. The fix: add errorCategory, isRetryable, and a human-readable description telling Claude what to do next.
Include errorCategory: transient | validation | permission
Include isRetryable: true/false boolean
Error message = "Order not found—try get_customer to search by phone" not "failed"
Anthropic docs: "include what went wrong AND what Claude should try next"
Q10 · Transient vs. business errors
Distinguish retriable vs. non-retriable at the tool level A
Business errors (wrong order ID, expired return window) will never succeed on retry — wasted turns. Technical errors (503, timeout) will.
DO
  • Return retriable: false + customer-friendly reason for business errors
  • Let model use that message directly in response
DON'T
  • Retry business errors (can never succeed)
  • Parse error text with few-shot examples to guess retryability
  • Add pre-flight eligibility tool (adds a round-trip every time)
02 Escalation Logic & Human Handoff Q2, Q5, Q8, Q11, Q12, Q14 ▶
Q11 · When to escalate
Use semantic triggers, not mechanical ones A
Rules engines, turn-count limits, and sentiment scores all miss cases. The model should escalate when: customer requests it, issue requires policy exceptions, or no meaningful progress is possible.
Turn-count triggers are too blunt — may escalate too early or too late
Sentiment analysis is a proxy, not the real signal
Rules engines can't enumerate every valid escalation scenario
Q12 · Guaranteeing resolution
For hard guarantees: use orchestration layer, not prompt instructions A
Model instructions can fail (3% rate). To guarantee every session ends with resolution or handoff, wrap the agent loop: if it terminates without resolution, programmatically call escalate_to_human from the orchestration layer.
System prompt instruction alone = not a guarantee (model can deviate)
80%-turns hook = premature escalation on most sessions
Two-agent split = different budgets but same problem if second agent also times out
Q14 · Policy-critical compliance
Hard business rules must be enforced in code, not prompt C
Refunds >$500 must escalate — 3% failure rate proves prompt alone is insufficient. Solution: intercept the tool call, check the amount, block + escalate if over threshold.
Few-shot examples and strong language still leave the model in control
Tool-level error message still requires model to act correctly
Only a pre-tool-use hook removes model discretion entirely
Q2 · Q5 · Handoff context
Pass structured summary, not raw transcript B / D
Human agents need actionable context, not 25+ turns of chat. Provide: customer ID, root cause, amounts, and recommended action. Raw transcripts are too long; verbatim complaint without diagnosis is too thin.
Q8 · Explicit escalation requests
Honor explicit "real person NOW" requests immediately A
When a customer explicitly demands a human, escalate at once — even without tool investigation. Do NOT gather context first, offer alternatives, or ask clarifying questions before escalating.
03 Agentic Loop & Context Management Q3, Q6, Q9, Q13, Q15 ▶
Q6 · How agentic decisions work
The model reasons from context — no pre-wired routing D
After each tool call, the result is added to conversation history. The model then reasons about what to do next. There is no decision tree, no pre-planned sequence, and no orchestration layer routing based on status fields.
Q13 · Context not passed between turns
Stateless API — always pass full conversation history B
If the agent "forgets" earlier turns (re-asks for name), the conversation history is not being passed in subsequent API requests. Claude has no memory between calls — all state lives in the messages array.
Not a prompt instruction issue; not a tool state issue
No default memory limit — if history is passed, Claude uses it
Q9 · Tool output bloat
Trim tool outputs to only what the task needs D
When lookup_order returns 40+ fields but you only need return eligibility, strip to relevant fields before adding to context. Saves tokens, reduces distraction.
Don't summarize to prose — structured data is more reliable for downstream actions
Vector DB retrieval adds latency and complexity for a simple trim
Q3 · Frustrated customer, solvable problem
Acknowledge + offer choice; don't force either path A
If you've verified the issue is resolvable, acknowledge frustration, inform the customer it can be fixed now, and let them choose: resolve immediately OR escalate. Don't process silently; don't force escalation.
Q15 · Backend failure mid-resolution
Explain situation honestly; offer escalation or retry D
If you can explain billing and verify eligibility but can't process the refund (timeout), don't claim it's done and don't escalate immediately. Explain the system issue and offer the customer options.
04 Tool Selection & Schema Design Q4, Q19, Q22, Q24, Q26, Q28, Q30 ▶
Q4 · Semantic overlap between tools
Fix overlapping tools by consolidating, not routing D
When two tools do similar things (issue_credit vs. process_refund), the model will keep getting confused regardless of examples. Merge them into one tool with a parameter. Eliminate the ambiguity structurally.
Few-shot examples reduce but don't eliminate the overlap problem
Sub-agents with routing adds complexity without fixing root cause
Deferred loading reduces tool count but overlapping tools still conflict when loaded
Q22 · Q26 · Reliable JSON schema compliance
Use tool_use / function calling — not prompt formatting instructions C / A
For strict schema compliance, define a tool whose input schema IS your target schema. Extract data from the tool_use response. This is structurally enforced, not prompt-enforced.
RELIABLE
  • Define tool with exact JSON schema as input params
  • Extract from tool_use response block
UNRELIABLE
  • Prompt instructions + regex parsing
  • Two-call text-then-format approach
  • Response prefilling with {
Q19 · Schema can't express "no data"
Use null (not empty array) for missing data; add enum values for ambiguous cases B
Empty array [] = "no pros" (valid). null = "pros not determinable from text" (distinct). For ambiguous sentiment, add "unclear" to the enum — don't let the model guess.
Q24 · Open-ended enum values
When new values keep appearing: free-form string + post-processing normalization D
If your enum keeps growing (property types: studio, loft, duplex, tiny house…), change to free-form string and normalize downstream. Adding "other" loses specificity; expanding enum never ends.
Q28 · Enforcing tool call order
Force first tool with tool_choice; run enrichment in subsequent turns D
When Tool B depends on Tool A's output, use tool_choice: {"type": "tool", "name": "extract_metadata"} for the first call. Then run enrichment tools in the next turn with the extracted data available.
Q30 · Guaranteed structured output, unknown doc type
Use tool_choice: "any" with all extraction tools defined B
"any" forces a tool call without specifying which one — the model picks the right tool for the document type. "auto" allows text responses; a single unified schema loses type specificity.
05 Batch Processing & Extraction Pipelines Q16, Q17, Q18, Q20, Q21, Q23, Q25, Q27, Q29 ▶
Q16 · Batch vs. real-time for large jobs
Iterative batch: submit all → fix failures in successive batches D
For 50k docs in 2 weeks: batch API (50% cheaper) + submit all first, then fix failed extractions in subsequent batches with refined prompts. Don't sample-then-batch (extra round-trip); don't use real-time (no cost benefit).
Q25 · SLA with batch API
Match batch frequency to SLA window minus processing time C
SLA = 30 hours. Batch API = up to 24h. Submit every 4 hours → worst case: doc arrives just after batch submission, waits 4h, processes up to 24h = 28h total. End-of-day batching (24h+ wait) risks missing SLA.
FrequencyWorst-case waitMeets 30h SLA?
End of day24h wait + 24h processing = 48h✗ No
Every 6h6h + 24h = 30h⚠ Borderline, no buffer
Every 4h4h + 24h = 28h✓ Yes (2h buffer)
Q17 · Recurring extraction failures
Fix systematic patterns with few-shot examples, not post-processing A
When 23% of errors share one root cause (informal measurements), add few-shot examples showing correct handling. Pattern matching in post-processing doesn't teach the model; fine-tuning on 847 samples is overkill for a prompt-fixable issue.
Q18 · Valid JSON but missing required fields
Show varied-format examples in few-shot, not schema changes or retry B
If content exists in the doc but the model misses it (citations in bibliographies, methodology in intro), show examples with each layout variant. Retry without new context = same failure. Regex post-processing is brittle.
Q20 · High-confidence extractions with errors
Stratified random sampling: catches unknowns AND measures trends A
Heuristic rules (flag comparison tables) only catch known error patterns. Random sampling catches novel patterns AND gives you a statistically valid error rate to measure improvement over time. Both serve different purposes — sampling is the sustainable primary strategy.
Q21 · Validation failures on retry
Include the specific validation error in the follow-up request B
Retrying without modification → identical failure. Send a follow-up: include the Pydantic error message, ask model to correct its output. The model can fix "expected float, got '2 to 3'" when it knows what's wrong.
Q23 · Routing semantic errors (not schema failures) to review
Field-level confidence scores + calibrated thresholds C
Semantic errors (duration in quantity field) pass validation. Route based on model-reported field confidence, calibrated against a labeled validation set. Empty-field routing misses the plausible-wrong-value problem entirely.
Q27 · Mathematical consistency (invoice totals)
Extract both stated and calculated totals; flag mismatches for human review D
Add stated_total and calculated_total fields. Flag discrepancies for review — don't auto-adjust (can't tell if OCR or model error), don't use a second model (expensive, same uncertainty).
Q29 · Routing to human review (limited capacity)
Route on model-indicated uncertainty, not entity type D
With only 5% review capacity, route where the model signals low confidence or source docs are ambiguous/contradictory. Routing by entity type (financial figures) wastes capacity on confident, correct extractions.
06 Claude Code — Config, Workflows & Memory Q31, Q32, Q33, Q34, Q35, Q36, Q37, Q38, Q39, Q40, Q41, Q44, Q45 ▶
Q31 · MCP server scopes
Project-scoped → .mcp.json · User-scoped → ~/.claude.json C
Shared team server: .mcp.json (committed to repo, all devs get it). Personal/experimental server: ~/.claude.json (user-level, not shared).
Q32 · Large-scope refactor
Plan mode first for multi-file, breaking-change migrations A
45 files, 3 breaking changes = explore first, then implement. Plan mode maps usage patterns before touching code. Running tests and fixing failures reactively misses the architectural picture.
Q33 · Task-specific checklists
Slash commands for context-specific workflows, not CLAUDE.md C
Code review checklist doesn't apply to all tasks. Put it in a /review slash command invoked only when reviewing. CLAUDE.md is always-on — it pollutes general coding sessions with review-specific guidance.
Q34 · One-off pattern following
@ reference the actual files; don't describe patterns in prose C
For a one-off task with 3 known pattern files, use @db_utils.py @error_handlers.py @audit_logger.py directly in the prompt. Natural language descriptions lose specificity; CLAUDE.md is for persistent conventions.
Q35 · Hard constraint vs. preference vs. automation
Match enforcement mechanism to requirement strength C
RequirementRight mechanism
Never edit db/migrations/permissions.deny — hard block
Prefer custom loggerCLAUDE.md — preference/guidance
Auto-format TS after editPostToolUse hook — automation
Q36 · Unknown requirements
Ask Claude to surface unknowns before implementing D
When you're new to a domain (production caching), have Claude interview you about considerations you haven't thought of — cache invalidation, failure modes, consistency. Starting minimal and patching is reactive; TBD specs are better but less interactive.
Q37 · Team-shared slash commands
Team skill files go in .claude/skills/ committed to version control B
For shared, versioned workflows: .claude/skills/migrate-component/SKILL.md at project root, committed to git. Per-machine files in ~/.claude/skills/ aren't shared and drift over time.
Q38 · Single-function change
Simple, well-scoped changes → direct execution D
Adding one conditional to one function needs no plan mode, no extended thinking. Overhead of planning exceeds the complexity of the task. Reserve plan mode for multi-file, architectural changes.
Q39 · Iterating on generated code
Give a concrete failing test case, not a vague re-description D
Provide input with null values + expected output → ask Claude to fix it. "Think harder" and full rewrites lose context of what already works. Manual edits diverge from Claude's understanding.
Q40 · MCP prompts in Claude Code
MCP prompts surface as slash commands: /mcp_servername__promptname C
MCP prompts are not auto-injected, not in the tool registry, not @-mentionable resources. They become slash commands you explicitly invoke.
Q41 · Selective standard application in monorepos
@import only the relevant standards per package B
Each package's CLAUDE.md uses @imports to pull in only the standards that apply (e.g., auth package imports security-rules.md + testing-patterns.md, not api-conventions.md). Maintainers know their domain.
Q44 · Inconsistent convention following
First diagnostic: check /memory to confirm CLAUDE.md is loaded B
Random inconsistency across sessions often means the memory file isn't loading. Run /memory to see which files are included. Don't strengthen language or add examples until you've confirmed the file is being read.
Q45 · Architectural decision with trade-offs
Plan mode for decisions affecting infrastructure/architecture B
WebSockets vs SSE vs polling have different infra requirements. Explore trade-offs, get team buy-in before implementing. Choosing the "simplest" option first creates rework when requirements shift.
07 MCP Servers & Multi-Agent Architecture Q46, Q47, Q48, Q51, Q54, Q46 ▶
Q46 · Multi-server tool access
All MCP tools are available simultaneously from the start B
Tools from all configured MCP servers are discovered at connection time and available simultaneously. The agent doesn't pick one server per turn; no sequential routing; no per-turn specification needed.
Q47 · Q57 · Parallel exploration branches
Use fork_session to explore multiple approaches from shared context B
When you want to explore two approaches (microservice vs. in-place refactor, E2E vs. snapshot tests) independently, fork the session. Both branches start from the same analysis, no context duplication needed.
Q48 · MCP tool ignored in favor of built-in
Expand MCP tool descriptions — the model picks based on description quality B
"Analyzes dependency graph" loses to Grep's detailed description. Expand to: "Builds dependency graph showing direct imports, transitive dependencies, and circular dependency cycles across the entire codebase." Don't remove Grep; don't add routing rules.
Q51 · Reducing exploratory tool calls
Expose content catalogs as MCP resources A
When the agent makes 8-10 calls because it doesn't know what each server contains, expose catalogs as MCP resources (issue summaries, doc hierarchy, DB schemas). Agent can read the map before making targeted calls.
Q54 · Accessing external services (Jira)
Use an existing Jira MCP server, not curl or manual exports C
Integrate a Jira MCP server for discoverable, maintained tool interfaces. Custom MCP wrapper = unnecessary build. Curl via Bash = fragile, requires inline auth. Markdown exports = stale, manual.
08 Codebase Exploration & Context Management Q42, Q43, Q49, Q50, Q52, Q53, Q55, Q56, Q58, Q59, Q60 ▶
Q49 · Q60 · Finding unknown code
Grep for exact error text first; then read context C / C
Unknown error message across 12 services: Grep for the distinctive string ("SYNC_CONFLICT"), read matching files. Don't read READMEs speculatively; don't search error handler directories (too broad). Same for aliased functions: find all exposed names first, then grep each.
Q50 · Q55 · Resuming sessions with changed files
Resume + inform about specific changed files B / A
Don't start fresh (loses 1h of context). Don't resume silently (stale assumptions on changed files). Tell the agent exactly which 3 files changed for targeted re-analysis. Architecture understanding remains valid; only the changed files need refresh.
Q52 · Context degradation mid-investigation
Spawn subagents for parallel sub-investigations A
When the main agent is losing accuracy after 8 files (45 remain), spawn subagents for specific questions ("find all test files," "trace refund flow") while main agent preserves high-level coordination. Don't clear context; don't switch to Grep-only.
Q53 · Edit tool can't find unique match
Read → modify → Write when Edit fails on repetitive files A
If the file has repetitive docstrings and Edit can't find a unique match, load the full file with Read, insert the function, Write back. Don't use extremely long old_string (fragile); don't append to end (wrong location).
Q56 · Understanding large unknown codebase
Follow the call graph incrementally from entry points C
800+ files: Grep for auth entry points, read those, follow imports/function calls outward. Don't read all "auth" files indiscriminately (too broad). Don't ask the user to pick files (defeats the purpose). Parallel subagents add complexity without direction.
Q58 · When multi-phase workflow adds value
Multi-phase only for complex, judgment-heavy tasks — not mechanical ones D
Rename function = mechanical, well-defined → direct execution. Error handling improvements = requires judgment about what "good" looks like, many trade-offs → analyze → propose → implement with review. Complexity and subjectivity justify the overhead.
Q42 · Iterative algorithm development
Write tests first; share failing tests to drive iteration B
For complex algorithms with edge cases, tests are the ground truth. Write the suite (disconnected nodes, cycles, weighted edges, performance). Share failing test output with each iteration — concrete, unambiguous feedback.
Q43 · Interacting formatting issues
Fix interacting issues sequentially, verify each step C
When column widths affect date rendering which affects page breaks, fix one at a time and verify before moving on. Tackling all at once makes it hard to isolate regressions. Showing a correct example is good but doesn't guide Claude on the technical specifics.
Q59 · Caching layer, 15 files, 8k lines
Follow class hierarchy from base class, trace invalidation specifically D
Don't load all 15 files (context overflow). Don't Grep for "invalidate" in isolation (misses the interface). Find the base cache class via imports, understand the interface, then trace concrete invalidation implementations. Architectural reading order, not file-size order.
