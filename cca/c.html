Key Concepts
Select a domain to review all concepts and anti-pattern.
D1 · CONCEPT 1
Agentic loop
The core cycle: send request → inspect stop_reason → if tool_use, execute tool and return result → repeat until end_turn or max_tokens. The loop is driven entirely by the orchestration layer, not by Claude itself.
Highest-weight domain · 27%
D1 · CONCEPT 2
Hub-and-spoke coordinator pattern
Most common multi-agent architecture. A coordinator (orchestrator) agent routes tasks to specialized subagents. The coordinator holds the master plan and aggregates results; subagents execute atomic, scoped tasks.
Expect scenario questions on this
D1 · CONCEPT 3
Subagent context isolation
Subagents have their own isolated context window — they do not inherit the coordinator's context. The coordinator must explicitly pass any required information in each subagent call. A common trick question on the exam.
Common trick question
D1 · CONCEPT 4
Parallel execution
Claude can emit multiple tool_use blocks in a single response. The orchestration layer executes these in parallel and returns all results together in the next tool_result batch — reducing total latency for independent tasks.
Pattern
D1 · CONCEPT 5
PostToolUse hooks
Hooks that fire after a tool call completes. Used for data normalization (reshape output format), validation (schema compliance checks), and audit logging — all before results re-enter the model context. One of the 4 hook types.
High-frequency
D1 · CONCEPT 6
Escalation with confidence scoring
Agents should self-assess certainty after each step. When confidence falls below a defined threshold, escalate to human review or a higher-capability model rather than proceeding silently. Prevents cascading failures.
High-frequency
D1 · CONCEPT 7
4 stop_reason values
The four values you must know cold: end_turn (finished normally), tool_use (wants to call a tool), max_tokens (cut off by length), stop_sequence (custom stop triggered). The agent loop branches on each.
Exact values tested
✕ ANTI-PATTERN · D1
Prompt-only behavior enforcement
Relying on prompt instructions alone to enforce critical behaviors (e.g., "never delete files") is unreliable — prompts can be misinterpreted by the model. Use hooks and programmatic code-level controls instead. The exam always favors code over prompt for enforcement.
D2 · CONCEPT 1
MCP = Model Context Protocol
Anthropic's open standard for connecting Claude to external services, databases, and APIs. MCP servers expose tools that Claude can discover and invoke. Think of it as a standardized plugin system — the interface is consistent regardless of what the server provides.
Definition must-know
D2 · CONCEPT 2
Tool description quality
Clear, specific tool descriptions are the primary driver of correct tool selection. Good descriptions state: what the tool does, when to use it, what it returns, and — critically — what it does not do (disambiguation from similar tools).
High-frequency
D2 · CONCEPT 3
Limited tools per agent
Optimal tool count per agent is small and focused. More tools = higher selection confusion. The pattern: specialized agents with 3–5 tightly scoped tools outperform a single agent with 20+ tools. Split large toolsets into specialized subagents.
Design principle
D2 · CONCEPT 4
Structured error responses
Tool errors should return structured objects with three fields: isError (bool), isRetryable (bool — tells the agent if retrying makes sense), errorCategory (string — auth, network, validation, etc.).
Exact field names tested
D2 · CONCEPT 5
tool_choice configuration
'auto' — Claude decides whether to use a tool. 'any' — Claude must use at least one tool. Forced specific tool — Claude must use exactly the named tool. Use forced mode when you need guaranteed structured output via tool_use.
Enum values tested
D2 · CONCEPT 6
Single responsibility per tool
Each tool should do one thing well. Avoid combining retrieval + transformation + storage into one tool. The orchestrator is responsible for composing simple tools — the tools themselves stay atomic and independently testable.
Design principle
✕ ANTI-PATTERN · D2
Vague tool descriptions
A vague description like "gets data" causes Claude to guess — and guess wrong. The fix is specificity with explicit exclusions: "Retrieves customer order history by customer_id from the orders database. Returns last 90 days only. Does not handle returns or refunds."
D3 · CONCEPT 1
Claude Code = CLI tool
Claude Code is the command-line interface for developers interacting with Claude in coding workflows. Not a web UI — it's a terminal-native agent that reads and modifies your local codebase directly.
D3 · CONCEPT 2
CLAUDE.md hierarchy
Instructions cascade: project root CLAUDE.md provides global context → .claude/rules/*.md adds modular domain-specific overrides. Examples: testing.md, security.md, api-style.md. Each file owns one concern.
Hierarchy order tested
D3 · CONCEPT 3
Custom slash commands
Project-specific shortcuts defined in CLAUDE.md: /review (code review workflow), /test (test suite with analysis), /deploy (deployment pipeline). Version-controlled, portable across the team.
Pattern
D3 · CONCEPT 4
Plan mode vs direct execution
Plan mode: Claude outlines what it will do before acting — safer for destructive or irreversible operations, allows human review of the plan. Direct execution: Claude acts immediately — faster for low-risk, well-defined, reversible tasks. The choice hinges on reversibility.
High-frequency decision scenario
D3 · CONCEPT 5
Modular rules files
Splitting CLAUDE.md into multiple .claude/rules/*.md files improves maintainability and avoids monolithic config drift. Each file is scoped to one concern and can be updated independently without touching global config.
Best practice
✕ ANTI-PATTERN · D3
One giant CLAUDE.md
A single large CLAUDE.md becomes hard to maintain, hard to review, and creates context bloat. The exam-correct answer is always to use modular .claude/rules/*.md files scoped to specific domains of concern.
D4 · CONCEPT 1
Few-shot prompting
Provide 2–3 input/output examples for ambiguous or format-sensitive tasks. Examples teach the expected pattern more reliably than lengthy instructions alone. Especially effective for classification and structured extraction tasks.
High-frequency
D4 · CONCEPT 2
Structured output via tool_use
The correct way to get guaranteed JSON structure is to define a tool with a JSON Schema and set tool_choice to force it. Asking Claude to "return JSON" in a text prompt is unreliable — the model may add prose, markdown fences, or deviate from schema.
Most important anti-pattern flip
D4 · CONCEPT 3
JSON Schema fields
In tool schemas: required array lists fields Claude must always populate. Optional fields may be omitted. enum constrains allowed values. strict mode prevents additional properties outside the defined schema.
Schema mechanics tested
D4 · CONCEPT 4
Message Batches API
Processes large volumes of requests asynchronously at ~50% cost reduction. Best for: data enrichment pipelines, bulk classification, offline analysis. Not suitable for real-time user-facing interactions requiring low latency.
Cost/use-case tradeoff tested
D4 · CONCEPT 5
Validation-retry loops
After receiving model output, validate against schema/business rules programmatically. On failure, re-prompt with the validation error as context. Cap retries (typically 3) to avoid infinite loops. The error message itself is the key input for self-correction.
Reliability pattern
D4 · CONCEPT 6
Chain-of-thought (CoT)
For complex multi-step reasoning, instruct Claude to think step-by-step before answering. Use <thinking> tags to separate internal reasoning from the final answer when you need clean output extraction downstream.
Pattern
✕ ANTI-PATTERN · D4
JSON in text vs tool_use
Asking Claude to "respond with JSON" in a text prompt is unreliable and schema-unsafe. Always use tool_use with a JSON Schema and forced tool_choice for guaranteed structured output. One of the most frequently tested anti-patterns across all domains.
D5 · CONCEPT 1
Lost-in-the-middle effect
LLMs attend strongly to the beginning and end of the context window. Content placed in the middle is underweighted and effectively ignored. Critical instructions buried mid-prompt in long contexts are not reliably followed.
High-frequency
D5 · CONCEPT 2
Progressive summarization
Compress earlier conversation turns into a running summary. Inject the summary at the top of the next request; discard the original turns. Preserves semantic history without exceeding the context window.
Pattern
D5 · CONCEPT 3
Context extraction
Rather than inserting full documents, retrieve only relevant sections via embedding similarity or keyword search. Inject extracted sections near the start or end of the prompt to benefit from primacy/recency attention.
Pattern
D5 · CONCEPT 4
Escalation with confidence thresholds
Agents must define a numeric or qualitative threshold below which output is too uncertain to act on. At that threshold: pause, escalate to human or stronger model. Never silently propagate uncertain outputs downstream in the pipeline.
High-frequency
D5 · CONCEPT 5
Scratchpad files
Persist intermediate reasoning and state to an external store (file, DB, KV). Only the distilled final result re-enters the model context for the next call. Keeps context lean and provides auditability of reasoning steps.
Pattern
D5 · CONCEPT 6
Context window management
Combined strategy: token budgeting (usable_input = context_limit − max_output_tokens), sliding window (drop oldest turns after summarizing), priority ordering (system → task → retrieved context → history), selective extraction for large docs.
High-frequency
✕ ANTI-PATTERN · D5
Dumping entire documents
Inserting full documents into every request causes context bloat, triggers the lost-in-the-middle effect, and leaves insufficient headroom for output. Always extract relevant sections. Use RAG/embedding retrieval for large corpora.
Anti-Patterns
All 5 domain anti-patterns — know these cold. They appear as the wrong answers in scenario questions.
D1 · Agentic Architecture · 27%
What to avoid: Relying on prompts alone to enforce behavior (no-delete, no-external-calls, etc.)
Correct approach: Use hooks and programmatic code-level enforcement. Prompts can be overridden by model interpretation; code cannot.
D2 · Tool Design & MCP · 18%
What to avoid: Vague tool descriptions like "gets data" or "processes request"
Correct approach: Specific, scoped descriptions with explicit exclusions. Tell Claude what the tool does AND what it does not do.
D3 · Claude Code · 20%
What to avoid: One monolithic CLAUDE.md file that tries to cover everything
Correct approach: Modular .claude/rules/*.md files, each scoped to a single concern (testing, deployment, security, style).
D4 · Prompt Engineering & Structured Output · 20%
What to avoid: Asking Claude to "return JSON" in a text prompt
Correct approach: Use tool_use with a JSON Schema and forced tool_choice for guaranteed, schema-validated structured output.
D5 · Context Management & Reliability · 15%
What to avoid: Dumping entire documents into every request
Correct approach: Extract only relevant sections and inject them near the prompt boundaries. Use RAG for large corpora.
⬡ Master rule behind all anti-patterns
Programmatic enforcement > prompt-based guidance. Structured mechanisms — hooks, schemas, modular files, tool_use — always win over relying on the model to follow instructions. When the exam presents a scenario with a reliability or correctness problem, the answer that uses code / hooks / schemas over prompts is almost always correct.
