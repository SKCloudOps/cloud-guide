Domain 01
Agentic Architecture & Orchestration
27% exam weight  ·  Most Tested Domain
What This Domain Covers
How Claude-powered agents are built, how they loop, and how multiple agents coordinate. The agentic loop is the fundamental execution pattern: call the API → read stop_reason → execute tools if tool_use → terminate if end_turn. Multi-agent systems layer a hub-and-spoke topology on top, where a coordinator decomposes tasks and dispatches isolated subagents. This domain also covers session management (resume vs. fork), escalation design, and failure handling.
ⓘ
Exam Strategy: Every agentic loop question traces back to one rule — stop_reason is the ONLY reliable control signal. Iteration caps, text parsing, and sentiment checks are always wrong answers. For multi-agent questions, remember subagents never inherit coordinator context.
Key Terms
Core
Agentic Loop
The while-true cycle at the heart of every Claude agent: call API → check stop_reason → if tool_use execute tools & append results → if end_turn return final response. Loop control must use the structured API field, never parsed text.
stop_reason = "tool_use" → continue  |  "end_turn" → terminate
Coordinator Agent
The top-level orchestrator that receives the goal, decomposes it into subtasks, and dispatches each to a specialist subagent. Coverage gaps in the final output always trace back here — not to the subagents.
Also called: hub agent, orchestrator
Subagent
An agent invoked by the coordinator for a specific scoped task. Subagents never automatically inherit the coordinator's conversation history — all context must be explicitly passed in their dispatch prompt.
Also called: worker agent, spoke agent
Synthesis Agent
A specialist subagent that combines findings from other agents into a final output. It should receive pre-gathered content only — it does not gather data itself. Giving it search tools causes misuse.
Also called: aggregator agent, merge agent
Hub-and-Spoke Topology
The standard multi-agent pattern. One coordinator (hub) dispatches work to multiple independent subagents (spokes). Spokes run in parallel when the coordinator emits multiple Task() calls in a single response turn.
Key rule: no shared memory between spokes — each is isolated
Parallel Task() Dispatch
When independent tasks have no data dependencies, the coordinator should emit all Task tool calls in a single response. Total elapsed time equals the longest single task, not the sum of all tasks.
Use when: tasks are independent — no output of one feeds another
Hook (Pre / Post)
A programmatic intercept that runs on every matching tool call — before or after execution — regardless of model output. Used for deterministic enforcement (refund caps, identity gates). Unlike prompts, hooks cannot be bypassed by the model.
Hooks = deterministic  |  system-prompt rules = probabilistic
--resume <session-name>
Restores the complete conversation state of an interrupted session — all prior messages, tool results, and reasoning. Picks up exactly where the session left off without any lossy summarization.
Use when: continuing an interrupted session on the same thread
fork_session
Creates an independent branch from the current session state. The fork inherits all prior context but diverges independently. Changes in the fork don't affect the main session — ideal for exploring alternative hypotheses.
Use when: side investigation without polluting the main thread
Structured Error Context
When a subagent fails, it must return structured data: failure type, partial results already completed, unprocessed IDs, and isRetryable. Never suppress errors or return empty results marked as successful.
Worst exam answer: "return empty result marked as successful"
Escalation Criteria
Reliable escalation triggers are objective and programmatic: (1) customer explicitly requests human, (2) policy is ambiguous, (3) agent cannot make progress. Sentiment and self-reported confidence are not reliable triggers.
Never: escalate on sentiment alone — calm users may need humans too
Progress Detection
An agent stuck in a loop should detect lack of progress (same tool, same params, same error) and terminate with a clear failure report. A generous token budget or wall-clock timeout is the correct safety net — not a fixed iteration cap.
Anti-pattern: fixed iteration cap of 5 regardless of ticket complexity
Iron Rules — Always True on Exam
Rule 1stop_reason is the ONLY reliable loop control signal. Never parse assistant text for completion phrases.
Rule 2Subagents never inherit coordinator context. All required information must be explicitly passed in the dispatch prompt.
Rule 3Coverage gaps trace to the coordinator's decomposition — not the individual subagents.
Rule 4Compliance rules (money limits, identity gates) belong in tool code (hooks), never in system prompts.
Rule 5Independent tasks should be dispatched in parallel (single response, multiple Task() calls) to minimize latency.
Common Exam Traps
⚠ Trap
Using a fixed iteration cap (e.g., 5) as the primary loop stop mechanism — truncates valid multi-step resolutions mid-flight.
⚠ Trap
Assuming subagents automatically inherit the coordinator's message history — they never do.
⚠ Trap
Using sentiment analysis as the escalation trigger — sentiment ≠ complexity of the ticket.
⚠ Trap
Using Claude's self-reported confidence for escalation — LLM confidence is poorly calibrated and non-deterministic.
⚠ Trap
Blaming a subagent for coverage gaps in the final output — always blame the coordinator's task decomposition.
⚠ Trap
Using fork_session to resume an interrupted session — use --resume instead; fork creates a divergent branch.

Domain 02
Tool Design & MCP
~20% exam weight  ·  High Frequency
What This Domain Covers
How to design tools Claude can call effectively, and how the Model Context Protocol (MCP) structures external integrations. The critical insight: tool descriptions are the primary routing mechanism — more influential than system prompts or few-shot examples. MCP separates tools (actions) from resources (read-only data) and defines a two-scope configuration system: project-level vs. user-level.
ⓘ
Exam Strategy: When a question says an agent is "calling the wrong tool," the fix is always improve the tool description — not add system-prompt instructions. Know the .mcp.json vs. ~/.claude.json split cold.
Key Terms
Core
Tool Description
The primary routing mechanism for tool selection — more influential than system prompts or few-shot examples. A complete description includes: input formats, example queries, explicit use-case boundaries, and "Do NOT use for X — use Y instead" statements.
Root cause of misrouting: minimal or overlapping descriptions
allowedTools
A parameter that restricts which tools an agent or subagent can invoke. Used in CI/CD pipelines to prevent destructive operations (write, deploy, push). Also scopes synthesis agents to only their intended tools.
Rule: give each agent the minimum tools needed for its role
.mcp.json
Project-scoped MCP configuration file. Committed to version control and shared with the entire team via git. Team MCP servers (Jira, GitHub, Slack) go here. Credentials use ${ENV_VAR} expansion — never hardcoded.
Rule: team tools → .mcp.json (version-controlled)
~/.claude.json
User-scoped MCP configuration file. Personal only — not shared via git, invisible to teammates. Personal tools and preferences go here. New team members will NOT receive these settings.
Rule: personal tools → ~/.claude.json (not shared)
MCP Resources vs Tools
Resources = read-only data sources (files, databases, APIs) exposed to the model for context. Tools = callable actions with side effects (write, notify, update). Side-effecting operations must be exposed as tools, not resources.
Resources = read-only context  |  Tools = side-effecting actions
Glob / Grep / Read / Bash
Glob = find files by name/path pattern. Grep = search file contents. Read = open a known file. Bash = run shell commands (tests, lint). Use dedicated tools first; Bash only when no dedicated tool exists for the operation.
Correct order: Glob → Grep → Read (for unknown codebases)
Tool Granularity
Each tool should have a single, clear purpose. Prefer separate focused tools (update_status, send_notification, log_interaction) over a single god-tool with an action enum. Focused descriptions give the model clear selection signals.
Anti-pattern: manage_ticket(action: enum) — hides real input shapes
Optimal Tool Count
Anthropic guidance recommends 4–5 tools per agent as the optimal range. Too many tools dilute description clarity; too few limits capability. Synthesis agents in particular should be scoped to only their synthesis-specific tools.
Anti-pattern: raw SQL access — unbounded surface, no enforcement
${ENV_VAR} Credential Expansion
MCP server credentials (API keys, tokens) must use environment variable expansion in config files — never hardcoded strings. This keeps secrets out of version-controlled files while keeping configuration shareable.
Never: hardcode API keys in .mcp.json (version controlled)
Decision Table — Which Config File?
ScenarioCorrect FileReason
.mcp.jsonTeam Jira/GitHub/Slack serversShared via git; all developers get the same tools without manual setup
~/.claude.jsonPersonal database GUI, private API keyNot version-controlled; invisible to teammates; personal preference
Hardcoded key in .mcp.jsonNeverCredentials exposed in version history; use ${ENV_VAR} expansion instead
Team tool in ~/.claude.jsonNeverNew team members won't receive it; defeats the purpose of team standardization
Common Exam Traps
⚠ Trap
Fixing tool misrouting by adding system prompt instructions — tool descriptions are more influential and are the correct fix.
⚠ Trap
Putting team MCP servers in ~/.claude.json — they won't be shared with new team members.
⚠ Trap
Exposing write/notify/update actions as MCP resources — side-effecting operations must be tools, not resources.
⚠ Trap
Giving a synthesis agent search tools — allowedTools should restrict it to synthesis-only operations.

Domain 03
Claude Code Workflows & Configuration
~22% exam weight  ·  High Frequency
What This Domain Covers
How Claude Code is configured, deployed, and extended. The most-tested concept is the CLAUDE.md hierarchy: project-level config (committed to git, shared with team) vs. user-level config (personal, invisible to teammates). This domain also covers headless CI/CD operation (the -p flag), Skills and slash commands, Plan Mode, and Claude Code hooks.
ⓘ
Exam Strategy: If a question asks how to share Claude Code configuration with a team, the answer always involves the project-level CLAUDE.md (committed, in git). If it asks how to run Claude Code in CI/CD without interaction, the answer is always the -p flag — not CLAUDE_HEADLESS=true (doesn't exist) or --batch (doesn't exist).
Key Terms
Core
CLAUDE.md Hierarchy
./CLAUDE.md = project-level, committed to git, shared with the whole team. ~/.claude/CLAUDE.md = user-level, personal only, not shared via git. Standards that new developers must receive go in project-level config.
Most-missed: user-level config is invisible to teammates
-p Flag (Headless / --print)
Makes Claude Code non-interactive: processes the prompt, prints output to stdout, and exits. Required for all CI/CD pipelines. The only documented headless mechanism — not CLAUDE_HEADLESS=true (doesn't exist) or --batch (doesn't exist).
Only correct syntax: claude -p "your prompt"
Plan Mode
Claude explores and proposes a plan before making any changes. Human reviews and approves, then execution begins. Use for: large-scale architectural changes, decisions across many files, multiple valid approaches. Not for simple reversible operations.
Key phrase: "explore codebase and design approach before committing"
.claude/rules/ + glob frontmatter
Rule files in .claude/rules/ apply globally unless given a YAML frontmatter glob: globs: ["api/**/*"]. Without a glob, rules activate for all files — a common misconfiguration that causes irrelevant suggestions.
Use when: conventions apply to specific file types regardless of directory
context: fork (Skills)
A skill frontmatter option that runs the skill in an isolated subagent. Only the final result returns to the main conversation — verbose intermediate output stays isolated and doesn't pollute the main context window.
Use when: skill output is verbose and would cause context pollution
argument-hint (Skills)
A skill frontmatter option that prompts the user for required parameters before the skill executes. Prevents skills from running with missing arguments and producing incorrect or incomplete results.
Use when: skill needs inputs like service name or environment target
Claude Code Hooks
Programmatic hooks in Claude Code: pre- and post-execution intercepts for enforcing team standards, audit logging, and business rules deterministically. Bypass-proof — execute on every matching tool call regardless of model output.
Use for: enforcing non-negotiable standards regardless of model output
allowedTools in CI
In GitHub Actions / GitLab CI, restrict Claude Code to read-only tools (Glob, Grep, Read, Bash for tests/lint). Prevent destructive operations by excluding write, deploy, push tools from the allowedTools list entirely.
CI safety: read-only tools only — never write/push in automated pipelines
Two-Pass Code Review
For large PRs (14+ files), a single-pass review causes attention dilution and contradictory feedback. Fix: Pass 1 = per-file analysis (parallelizable). Pass 2 = cross-file integration pass for data flow and shared interfaces.
Never: "use a larger context window model" — doesn't fix attention quality
Independent Review Instances
When reviewing code for multiple concerns (security, performance, style), use separate review instances per concern without sharing each other's context. Prevents cross-contamination and contradictory suggestions.
Anti-pattern: one agent reviewing all concerns — produces contradictions

Domain 04
Prompt Engineering & API Features
~18% exam weight  ·  Medium-High
What This Domain Covers
How to elicit reliable, structured output from Claude and how to choose the right API feature for each use case. Key topics include guaranteeing structured output with tool_choice: "any", preventing fabrication with nullable schema fields, using few-shot examples for decision calibration, the Message Batches API tradeoffs, and understanding when extended thinking adds value.
ⓘ
Exam Strategy: tool_choice: "any" is the only guaranteed mechanism for structured output — system-prompt JSON instructions are probabilistic. The Message Batches API is correct only for non-blocking workflows. Few-shot examples fix calibration issues, not compliance rules (hooks do that).
Key Terms
Core
tool_choice: "any"
Forces Claude to call a tool (producing structured JSON) rather than returning conversational text. The only guaranteed mechanism for structured output. tool_choice: "auto" allows text responses. System-prompt JSON instructions are probabilistic only.
"any" = guaranteed structured JSON  |  "auto" = maybe text
Nullable Schema Fields
Making schema fields type: ["string", "null"] prevents fabrication when source documents don't contain that data. Required fields force the model to invent values; nullable fields let the model return null honestly.
Cause of fabrication: required schema fields when data may be absent
Few-Shot Examples
Examples that teach decision boundaries and output format. Effective for: escalation calibration, tool selection in ambiguous cases, output format consistency. Not for compliance enforcement (use hooks) or before fixing tool descriptions (fix root cause first).
Not for: financial compliance rules — those require programmatic hooks
Message Batches API
Asynchronous batch processing with 50% cost savings but up to 24h processing time and no SLA guarantee. Correct for non-blocking workflows (overnight reports, weekly audits). Never for blocking workflows where users wait for results.
Blocking = real-time only  |  non-blocking = batch candidate
Extended Thinking
Claude's extended thinking mode for complex reasoning tasks. When to enable: multi-step logical problems, code architecture, mathematical reasoning. Tradeoffs: higher latency, higher cost, larger token budget required. Do not use for simple factual tasks.
Use when: problem genuinely benefits from step-by-step internal reasoning
Resolvable vs Unresolvable Errors
Resolvable: data exists but is formatted wrong — retries can fix it. Unresolvable: data doesn't exist in the source — retries waste API calls. When retries consistently fail for some documents, the fix is schema redesign, not increasing retry count.
Exam trap: "increase retries from 3 to 10" — almost always wrong
Self-Review Limitation
Reviewing code in the same session that generated it is less effective — the model retains its generation reasoning and confirmation-biases toward its own decisions. Independent review = different session, no prior generation context.
Fix: always use a fresh session for code review — different context
System Prompt Scope
System prompt instructions are guidance, not enforcement. Prompt injection, long contexts, or unusual user phrasing can cause the model to ignore system-prompt rules. Compliance-critical rules must live in tool code (hooks), not in the system prompt.
Rule: probabilistic guidance only — can be bypassed by the model
Prompt Chaining vs Dynamic Decomposition
Prompt chaining: all steps known upfront, fixed pipeline (PR review with known passes). Dynamic decomposition: scope unknown, plan adapts to discoveries (exploring unfamiliar legacy codebase). Match the pattern to the task structure.
Use dynamic: when discoveries determine the next step
Structured Output — Comparison
✗ Probabilistic — Don't Rely On
System prompt: "Always respond in JSON format"

Instructions for JSON shape in the prompt

tool_choice: "auto" (may return text instead)

Response parsing with regex fallbacks
✓ Guaranteed — Use This
tool_choice: "any" (forces a tool call)

Tool schema with precise field definitions

Nullable fields for optional data (prevents fabrication)

Tool result as structured data, not parsed text

Domain 05
Context & Memory Management
~13% exam weight  ·  Medium
What This Domain Covers
How information is preserved, degraded, and managed across long conversations and large inputs. Key failure modes include the Lost in the Middle phenomenon, Progressive Summarisation Loss of exact numerical values, and how to handle conflicting sources and multi-match disambiguation. The /compact command and scratchpad files are the primary mitigation tools in Claude Code.
ⓘ
Exam Strategy: For "Lost in the Middle" questions, the correct fix is always restructure input placement (key content at top/bottom) — never "use a larger model." For numerical precision loss, the fix is a case-facts block outside the summarisation pipeline, never "summarise more aggressively."
Key Terms
Core
Lost in the Middle
Models reliably attend to content at the beginning and end of long inputs but miss findings in the middle. A larger context window does not fix this. Fix: place key findings at the top + use explicit section headers throughout the document.
Never correct: "use a larger model" or "increase context window"
Progressive Summarisation Loss
Compressing old conversation turns loses exact numerical values (amounts, order IDs, dates) into vague summaries. Critical transactional data must be preserved in a persistent case-facts block outside the summarisation pipeline.
Never correct: "summarise more aggressively"
Case-Facts Block
A structured block of critical data (order IDs, amounts, dates, issue types) stored outside the summarisation pipeline and included verbatim in every prompt. Prevents numerical precision loss across long multi-issue sessions.
Format: JSON with exact values — never prose summaries
/compact
A Claude Code command that reduces context window usage when the window is filling up and answers become vague or reference "typical patterns" instead of specific earlier findings. Pair with scratchpad files to preserve key discoveries.
Symptom: Claude references "typical patterns" instead of specific findings
Disambiguation Rule
When a lookup returns multiple matches, the agent must ask for additional identifying information (email, phone, account number). Never select by heuristic (most recent, first result, most orders). Heuristic selection risks acting on the wrong account.
Rule: always ask — never guess when multiple matches exist
Conflicting Sources
When credible sources disagree, present both values with source attribution and note the discrepancy. Never: average (fabricates a number neither source supports), use the more recent (recency ≠ accuracy), or omit (losing information).
Rule: preserve the conflict with attribution — never resolve arbitrarily
Scratchpad Files
External files used by Claude Code to persist key discoveries across context boundaries. When the context window fills and /compact is run, scratchpad files ensure important findings aren't lost in the compression.
Use alongside: /compact — one compresses, one preserves
Context Window vs Attention
A larger context window size does not improve attention quality. Increasing the window just lets the model ignore more content simultaneously. The fix for attention problems is input structure (placement, headers), not window size.
Larger window ≠ better attention to middle content
Iron Rules — Always True on Exam
Rule 1Exact numerical values (amounts, IDs, dates) must be preserved in a case-facts block outside the summarisation pipeline.
Rule 2A larger context window does NOT fix "Lost in the Middle." Restructure input placement instead.
Rule 3When multiple records match a lookup, always ask for disambiguation — never select by heuristic.
Rule 4When sources conflict, present both with attribution. Never average, never pick arbitrarily.
Rule 5Summarise more aggressively is always the wrong answer for numerical precision loss.
Common Exam Traps
⚠ Trap
Fixing "Lost in the Middle" by upgrading to a larger model — window size doesn't fix attention quality.
⚠ Trap
Fixing numerical precision loss by summarising more aggressively — more compression causes more loss, not less.
⚠ Trap
Resolving conflicting sources by averaging the values — averaging fabricates a number neither source supports.
⚠ Trap
Selecting the most recent matched record when multiple exist — recency is a heuristic that risks acting on the wrong account.
