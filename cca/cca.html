Coordinator Agent
The top-level orchestrator in a multi-agent system. It receives the high-level goal, decomposes it into subtasks, assigns each subtask to a specialist subagent, and aggregates the results. When output is incomplete or wrong, the root cause is almost always here — not in the subagents.
Also called: orchestrator, hub agent
Synthesis Agent
A specialist subagent whose role is to combine, merge, and summarise findings from other agents (web search, document analysis, etc.) into a final coherent output. It should only receive pre-gathered content — it does not gather data itself.
Also called: aggregator agent, merge agent
Subagent
Any agent invoked by the coordinator to handle a specific, scoped piece of work (e.g., "search for AI in music," "analyse contract PDF"). Subagents never automatically inherit the coordinator's conversation history — all context must be explicitly passed in their prompt.
Also called: worker agent, spoke agent
Agentic Loop
The while-true cycle that drives a single agent: call API → check stop_reason → if tool_use execute tools & loop again → if end_turn return response. The loop's control signal is always stop_reason, never parsed text.
Tested in: D1 — Agent Loop (27% weight, most tested)
Hub-and-Spoke
The standard multi-agent topology where one coordinator (hub) dispatches work to multiple independent subagents (spokes). Spokes run in parallel when the coordinator emits multiple Task() calls in a single response turn. Results return to the hub for synthesis.
Key rule: subagents are isolated — no shared memory between spokes
Hook
A programmatic intercept that runs on every matching tool call — before or after execution — regardless of what the model outputs. Used for deterministic enforcement of business rules (refund caps, identity verification gates). Unlike prompt instructions, hooks cannot be bypassed by the model.
Key distinction: hooks = deterministic  |  prompts = probabilistic
D1 — Agentic Loop
27% Exam WeightMost Tested DomainD1 Agentic Architecture

What it is: The agentic loop is the fundamental execution pattern of every Claude-powered agent. An agent calls the Claude API, reads the stop_reason field in the response, and uses that signal — and only that signal — to decide whether to continue (execute tools and call API again) or terminate (return the final response).

Why it matters: The loop pattern is the backbone of all agentic systems. Getting the control signal wrong (parsing text, using iteration caps as primary stop) leads to infinite loops, premature termination, or lost tool context. Every more complex multi-agent concept builds on this foundation.

Core control signal
stop_reason — the ONLY reliable way to know if Claude is done. tool_use = keep going, end_turn = finished.
Key concepts covered
stop_reasontool_useend_turnmessage historyprompt chainingdynamic decomposition
Common exam traps
Parsing text for “I’m done” • Using iteration cap as primary stop • Not appending both assistant turn AND tool results before next API call
D1.1 — The Agentic Loop: stop_reason is the only reliable signal27% WEIGHT · MOST TESTED
Core Rule
After every API call, read stop_reason. "tool_use": execute tools, append results, call API again. "end_turn": Claude finished, return response. Never parse assistant text, never cap iterations as primary stop.

Why each distractor fails — elimination logic

Distractor optionWhy it fails
Parse text for "I'm done" / "task complete"Claude never guarantees specific phrasing. Model updates silently change output format. Non-deterministic — one prompt variation breaks it.
Iteration cap (e.g. 10) as primary stop mechanismTerminates mid-task when agent may be 80% through a complex operation. Valid ONLY as a safety backstop, never as primary signal.
Check whether response contains assistant textClaude often returns BOTH text AND tool_use blocks in one response. This would terminate the loop prematurely.
Check stop_reason fieldExplicit, guaranteed, version-stable API contract. "tool_use" = more work. "end_turn" = Claude done. Never ambiguous.
Correct implementation — study this pattern
def run_agent(user_message): messages = [{"role": "user", "content": user_message}] while True: resp = client.messages.create( model="claude-sonnet-4-20250514", tools=TOOLS, messages=messages) ``` # THE ONLY reliable loop control signal if resp.stop_reason == "end_turn": return resp.content[0].text if resp.stop_reason == "tool_use": results = [] for block in resp.content: if block.type == "tool_use": output = execute_tool(block.name, block.input) results.append({ "type": "tool_result", "tool_use_id": block.id, "content": output }) # MUST append BOTH turns to maintain history messages.append({"role": "assistant", "content": resp.content}) messages.append({"role": "user", "content": results})

Critical: you must append both the assistant's full response AND the tool results as a user turn before the next API call. Omitting either causes Claude to lose context of what tools it called and what they returned.

Exam tip: "Loop runs forever" → not detecting end_turn. "Terminates too early" → checking text content instead of stop_reason. "Agent loses tool context" → not appending both turns to message history each iteration.
D1.6 — Task Decomposition: Prompt Chaining vs DynamicHIGH FREQ
Decision Rule
Prompt chaining: all steps known upfront, fixed (PR review with known passes). Dynamic decomposition: scope unknown, plan adapts to discoveries (exploring unfamiliar legacy codebase). Match the pattern to the task structure.

The 14-file / 20-file PR review — official exam question (sample-12)

Single-pass review of many files produces: inconsistent depth, missed bugs, contradictory feedback (flags a pattern in file A, approves identical code in file B). Root cause: attention dilution when processing many files simultaneously.

The fix is always two-pass decomposition:

  • Pass 1: Analyze each file individually for local issues (parallelisable)
  • Pass 2: Separate cross-file integration pass examining data flow and shared interfaces
Wrong optionWhy it fails
Switch to larger context window modelWindow size does not fix attention quality. Model attends poorly to middle of large inputs regardless of window size.
Run 3 independent passes; flag consensus issues onlyConsensus filtering SUPPRESSES real bugs caught only intermittently. You find fewer issues, not fewer false positives.
Require developers to submit smaller PRsShifts burden to developers. Does not fix the review system architecture.
Per-file local pass then separate cross-file integration passDirectly addresses attention dilution. Each pass has focused scope. No contradictory feedback possible.
Exam tip: "Inconsistent depth" or "contradictory feedback across files" always signals the two-pass decomposition answer. Appears specifically in Claude Code CI scenario.
D1 — Multi-Agent Systems
27% Weight (shared D1)High FrequencyD1 Agentic Architecture

What it is: Multi-agent systems use multiple Claude instances working together. The dominant topology is Hub-and-Spoke: a Coordinator Agent receives the high-level goal, decomposes it into focused subtasks, dispatches each to a specialist Subagent (web search, document analysis, synthesis), and aggregates the results.

Why it matters: Multi-agent patterns appear in nearly every complex AI system. The exam tests whether you know where bugs originate (always the coordinator’s decomposition, not the subagents), how context flows (explicitly, never automatically), and what parallel execution actually means (multiple Task() calls in one turn).

Iron rules (always correct)
Subagents NEVER inherit coordinator context — must be explicitly passed. Coverage gaps = coordinator decomposition problem, not subagent problem.
Key concepts covered
coordinator agentsubagentsynthesis agenttask decompositionparallel Task()error propagationstructured error context
Common exam traps
Blaming subagents for coordinator decomposition errors • Assuming context inheritance • Returning empty result as “successful” (always the worst answer)
D1.2 — Multi-Agent Hub-and-Spoke: Rules That Never ChangeHIGH FREQ
Rule 1: Subagents NEVER inherit coordinator context
Every piece of information a subagent needs must be explicitly passed in its prompt. No automatic propagation. Any answer option suggesting "subagent inherits coordinator history" is always wrong — eliminate immediately.
Rule 2: Coverage gaps trace back to coordinator decomposition
If the output misses entire domains (e.g., misses music and film when researching "AI in creative industries"), the subagents executed correctly within their narrow assignments. The fix is always coordinator-level decomposition, not downstream agents.
Rule 3: Parallel execution = multiple Task calls in ONE response turn
One Task call then wait then another = sequential. Multiple Task calls emitted in the same response = parallel. The exam tests whether you know the mechanism.

The verify_fact tool scenario — 85% simple / 15% complex

85% of synthesis verifications are simple fact-checks. 15% require deeper investigation. All currently go through coordinator (2-3 round trips, +40% latency). Best fix:

D1.3 — Error Propagation in Multi-Agent SystemsHIGH FREQ
The Rule
When a subagent fails, return structured error context. The coordinator makes the recovery decision. Never: suppress errors, terminate the whole workflow from a subagent, return generic status strings.
OptionVerdictReasoning
Return structured context: type, query, partial results, isRetryableCORRECTCoordinator gets full picture. Can retry with different query, use partial results, or try alternative approach.
Retry internally with backoff, return “search unavailable” after exhaustionWRONGGeneric status hides failure type. Coordinator cannot retry with different query or use the partial results already found.
Return empty result marked as successfulWORST — eliminate firstSilent suppression. Coordinator proceeds as if data exists, producing silent incomplete output.
Propagate exception, terminate entire workflowWRONGOne timeout destroys all completed work from other subagents. Recovery impossible.

Timeout vs empty results — specific exam question

Search returned “0 results” for category A and “timeout” for category B. How to handle each?

  • “0 results” = Successful query, no matches. Valid information. Do NOT retry. Do NOT flag as error. Coordinator can annotate the gap.
  • “Timeout” = Access failure. Query never completed. Mark isRetryable=True. Coordinator can retry with simpler query or alternative approach.
  • Wrong answer trap: “Report both as failures requiring coordinator intervention.” This causes unnecessary retry of the valid empty-results query.
Exam tip: “Return empty result marked as successful” is always the worst answer. Eliminate it immediately on every error propagation question. It appears as a distractor in nearly every multi-agent error scenario.
D1 — Hooks & Escalation
Most Tested in D1High FrequencyD1 Agentic Architecture

What it is: Hooks are programmatic interceptors that fire on every matching tool call, independent of model output. They enforce business rules — refund caps, identity verification gates, audit logging — with 100% determinism. Prompts are probabilistic (the model may ignore them). Hooks are deterministic (the model cannot bypass them).

Why it matters: Financial and compliance rules cannot rely on probabilistic enforcement. A 12% failure rate on identity verification or an occasional $600 refund violating a $500 cap are production disasters. The exam consistently tests whether you know when to use programmatic gates vs. prompt instructions.

The core distinction
Hooks = deterministic (100% enforced, no exceptions) • Prompts = probabilistic (high compliance, but non-zero failure rate)
Key concepts covered
pre/post hooksprerequisite gatesescalation triggers$500 refund capidentity verificationsession management
Valid escalation triggers
Policy gap • Customer explicitly requests human • Defined business threshold exceeded. NOT sentiment, NOT self-reported confidence, NOT multi-concern requests.
D1.5 — Hooks: Programmatic Enforcement vs Prompt InstructionsMOST TESTED IN D1
The Fundamental Distinction
Financial, security, compliance rules → programmatic enforcement (hooks, prerequisite gates). Style preferences, soft guidelines → prompts. Prompts = probabilistic (model sometimes ignores). Hooks = deterministic (always enforce, cannot be bypassed by model output).

Canonical exam scenario

You explored a codebase for 2 hours yesterday. You made significant code changes overnight. Today you need to continue analysis. Options: A) –resume, B) fresh session + inject yesterday’s key findings summary, C) resume and re-read all files, D) fork_session.

Answer: B. Yesterday’s tool results reference files that no longer reflect reality. Resuming (A) gives stale context. Re-reading all files (C) wastes the entire context window repeating work. Fork_session (D) preserves the stale context.

D2 — Tool Design
~20% Exam WeightHigh FrequencyD2 Tool Design & MCP

What it is: Tool design governs how Claude selects and correctly invokes tools in agentic systems. The tool description is the primary routing mechanism — it is more influential than the system prompt or few-shot examples. Poor, minimal, or overlapping descriptions are the root cause of nearly every misrouting failure.

Why it matters: An agent that calls the wrong tool (e.g., get_customer instead of lookup_order) produces wrong results regardless of how good the rest of the system is. This domain also covers MCP server scoping (project vs user) and when to use each built-in tool (Glob, Grep, Read, Bash).

Tool description rule
Expand descriptions first — always the highest-leverage first fix for misrouting. Include: input formats, example queries, explicit boundaries (“do NOT use for X”), edge cases.
Key concepts covered

D2.1 — Tool Descriptions: Root Cause of MisroutingHIGH FREQ
The Rule
Tool descriptions are the primary routing mechanism — more than the system prompt, more than few-shot examples. Minimal or overlapping descriptions = misrouting. Expanding descriptions is ALWAYS the highest-leverage first fix.

Two-tool misrouting scenario — official exam question (sample-2)

get_customer (“Retrieves customer information”) and lookup_order (“Retrieves order details”) both accept similar identifiers. Agent calls get_customer for “check my order #12345” queries.

OptionVerdictWhy
A: Add 5-8 few-shot examples showing order queries to lookup_orderWRONG first stepAddresses symptoms, not root cause. Token overhead without fixing ambiguous descriptions.
B: Expand each tool description with inputs, examples, edge cases, boundariesCORRECTDirectly fixes root cause. Tool descriptions ARE the routing mechanism. Always the right first step.
C: Implement routing layer pre-selecting tools by keywordsWRONGOver-engineered. Bypasses LLM natural language understanding. Adds maintenance burden.
D: Consolidate into single lookup_entity toolWRONG first stepArchitectural change. May be valid later but not the right first response to a description problem.

What a complete tool description contains

  • Input formats: “Accepts customer_id (CUS-XXXXX) or email. Does NOT accept order IDs.”
  • Example queries: “Use for: account auth, account history, verify customer identity”
  • Explicit boundaries: “Do NOT use for order status queries — use lookup_order instead”
  • Edge cases: “Returns list if multiple customers match. Caller must disambiguate before proceeding.”
✗ Minimal description (causes misrouting)
get_customer: “Retrieves customer information” lookup_order: “Retrieves order details” # No input format, no examples, # no boundary between them # Model guesses incorrectly 15-40%
✓ Complete description (routes correctly)
get_customer: “Verifies customer identity. Input: customer_id (CUS-XXXXX) or email. Use for: account auth, account history. Do NOT use for order status queries. Use lookup_order when you have an order_id.” lookup_order: “Retrieves order by order_id. Use for: shipping, returns, status. Requires order_id, NOT customer_id or name.”

Tool quantity problem — synthesis agent misusing search tools

Synthesis agent has access to 12 tools (search, document loaders, synthesis tools) and frequently uses search tools instead of synthesizing.

  • Root cause: Too many tools (12 vs recommended 4-5) degrades selection reliability. Agents given tools outside their role tend to misuse them.
  • Correct fix: Restrict synthesis agent toolset to only synthesis-relevant tools via allowedTools scoping.
  • Wrong fix: “Add prompt instructions telling synthesis agent not to use search tools” — probabilistic, less reliable than scoped access.
Exam tip: Two-tool misrouting question → expand tool descriptions first. Multi-agent wrong-tool-use question → too many tools in scope, restrict via allowedTools. These are different problems.
D2.2 — Built-in Tools: Glob vs Grep vs ReadMEDIUM FREQ
The Tool Selection Rule
Glob = find files by name/path pattern. Grep = search file CONTENTS for a pattern. Read = inspect a specific known file. The correct order for unknown codebases: Glob first, then Grep, then Read. Never start with Read on an unknown codebase.
ToolPrimary purposeUse forNOT for
GlobFile path pattern matchingFinding files by name/extension: **/*.test.tsx, terraform/**Searching file contents for patterns
GrepContent search across filesFinding function usages, import patterns, specific strings inside filesFinding files by name or extension
ReadInspect a specific known fileOpening and reading a file whose path you already know (from Glob/Grep)Discovering what files exist in unknown codebases
WriteCreate or overwrite filesGenerating new boilerplate, writing test filesReading or discovering file content
BashExecute commandsRunning npm test, npm run lint, git commands (scope with allowedTools in CI)File discovery or content search when dedicated tools exist

Canonical exam question: find *.test.tsx files, then search for a function within them

The question gives two operations: (1) find all files matching *.test.tsx, (2) search for a function name within those files. Which tools?

  • Correct: Glob to find files matching **/*.test.tsx, then Grep to search for the function name within those files.
  • Wrong: “Use Bash with find and grep” — dedicated tools exist, Bash is not the right choice.
  • Wrong: “Use Read on every file in the repository” — extremely inefficient and defeats the purpose of specialized tools.
  • Wrong: “Use Grep for both” — Grep searches content, not file names. It cannot find files by extension pattern.
Exam tip: Exam gives two operations. Map each to its tool. File discovery by name = Glob. Content search = Grep. Specific file inspection = Read. Bash is for commands (tests, lint, builds) not file discovery.

context:fork for verbose skills — official question (d3-2)

You create a /analyze-codebase skill that performs comprehensive analysis and generates verbose output. After running it, Claude loses track of the original task.

Root cause: Verbose skill output accumulates in the main conversation context window, causing context pollution and attention dilution.

Fix: Add context: fork to the skill frontmatter. This runs the skill in an isolated subagent. Only the final summary returns to the main conversation. Verbose output stays isolated.

CI/CD headless mode — official question (sample-10)

Pipeline script runs claude “Analyze this PR for security issues” but job hangs indefinitely. Logs show Claude Code is waiting for interactive input.

OptionVerdictWhy
A: Add -p flag: claude -p “Analyze this PR…”CORRECT-p (–print) is the documented non-interactive flag. Processes prompt, outputs to stdout, exits without waiting.
B: Set CLAUDE_HEADLESS=true environment variableWRONGThis environment variable does not exist in Claude Code.
C: Redirect stdin from /dev/nullWRONGUnix workaround that does not properly address Claude Code command syntax.
D: Add –batch flagWRONGThis flag does not exist in Claude Code.

allowedTools restriction in CI — why it matters

In CI/CD pipelines, Claude Code must not be able to write files, push to git, or deploy. The correct fix is not a prompt instruction but allowedTools restriction:

claude -p –allowedTools "Read,Bash(npm run lint),Bash(npm test)" "Review this PR diff. Report ONLY confirmed issues…"

The allowedTools parameter explicitly restricts what commands Bash can run. Only npm run lint and npm test are permitted. Write, deploy, and git commands are blocked deterministically.

Preventing duplicate review comments on re-run — official question (d3-5)

After developers push fixes addressing review comments, re-running the review duplicates old comments plus new ones.

Fix: Include prior review findings in context and instruct Claude to report only new or still-unaddressed issues. Not: clear all previous comments (loses history). Not: only run on new files (misses cross-file issues).

Exam tip: For any Claude Code question: project-level config = shared. -p flag = headless CI. context:fork = prevent context pollution. Plan mode = architectural decisions with exploration needed first.

Duplicate comment prevention (Q48)

After developers push fixes, re-running reviews duplicates old comments. Correct fix: include prior review findings in context and instruct Claude to report only new or unresolved issues. Wrong fixes: scope to newest files only (misses persistent issues in unchanged files); string-match post-filtering (brittle, misses rephrased duplicates); different session IDs (doesn’t address duplication).

D3.5 — Batch API Architecture Constraint in Agentic ReviewsNEW · CRITICAL DISTINCTION
Why Agentic Reviews Cannot Use Batch (Q50)
The Message Batches API is fundamentally asynchronous. Claude processes the batch offline; you retrieve results later. Agentic code reviews require a real-time tool execution loop: Claude calls a tool (e.g., read file), receives the result mid-request, and continues reasoning. This loop is architecturally incompatible with batch processing — not just slow, but technically impossible.

Batch API decision matrix — Q51, Q52 patterns

PR style checksYESReal-time onlyDeveloper waits for result.
Overnight deep analysisNOBatchNo immediate latency requirement. 50% cost savings.
Weekly security auditNOBatchScheduled, no urgency.
Nightly test generationNO (results by morning)BatchResults needed by morning — batch 24h window fits.

False positive trust recovery — Q57 pattern

When one category (style, documentation) has 60% false positive rates, it poisons developer trust in ALL findings — even accurate security and bug detection findings.

Correct fix: temporarily disable high false positive categories. Keep only high-precision categories visible while improving the failing category’s prompts offline. The trust damage cannot be mitigated with confidence scores, disclaimers, or uniform strictness reduction.

Exam tip: The batch API constraint question (Q50) is tricky: the answer is NOT latency (too slow) — it’s the asynchronous architecture being incompatible with tool execution mid-request. Eliminate options that reference latency or missing correlation IDs as the primary constraint.

D4.1 — Structured Output: Nullable Fields and tool_choice:“any”HIGH FREQ
The Two Key Rules

Validation retry loops — when retries succeed vs fail

Your pipeline retries extraction on schema validation failure. For some documents, retries consistently fail because the information is absent from the source.

Key distinction the exam tests: retries only work for resolvable errors (format mismatches, structural output errors where data exists but is formatted wrong). Retries do NOT work for unresolvable errors (data simply does not exist in the source document).

  • Resolvable: model returned {“date”: “15/03/2026”} but schema expects ISO format. Retry with validation error in prompt = model can fix format.
  • Unresolvable: invoice has no purchase order number. No matter how many times you retry, the data is not there.
  • Wrong answer: “Increase retries from 3 to 10.” This wastes API calls on unresolvable cases.
Exam tip: When the exam says “retries consistently fail for some documents” — the answer is to distinguish resolvable from unresolvable errors, not to increase retry count.
D4.2 — Batch API: A Latency Decision, Not a Cost DecisionHIGH FREQ
The Decision Rule
The Message Batches API saves 50% on costs but has up to 24-hour processing with NO guaranteed SLA. Blocking workflows (user waits for result, pre-merge checks) MUST use real-time API. Non-blocking workflows (overnight reports, weekly audits) are batch candidates.

The official exam question (sample-11)

Two workflows: (1) blocking pre-merge check that must complete before developers can merge, (2) technical debt report generated overnight for next-morning review. Manager proposes switching both to Batch API for 50% savings. How to evaluate?

OptionVerdictWhy
A: Batch for technical debt reports only; real-time for pre-merge checksCORRECTPre-merge check is blocking — developer waits, no 24h acceptable. Overnight report has no time urgency.
B: Switch both to batch with status polling for completionWRONG“Often faster” is not acceptable for blocking workflows. No SLA means no guarantee.
C: Keep real-time for both to avoid batch result ordering issuesWRONGBatch results CAN be correlated using custom_id fields. Ordering is not actually an issue.
D: Both to batch with timeout fallback to real-timeWRONGAdds unnecessary complexity when the simpler solution is matching each API to its use case.

SLA calculation question — official question (d4-6)

Processing 500 documents/week. Batch takes up to 24h. SLA requires results within 30h of submission. How to schedule?

Answer: Submit documents in 4-hour windows. 30h SLA - 24h max batch processing = 6h buffer. Submitting in 4-hour windows ensures the batch completes with time to spare before the SLA.

  • Wrong: Submit all 500 on Monday morning (single large batch might take the full 24h, risking SLA breach for documents submitted late Monday).
  • Wrong: Use real-time API for all (loses 50% cost savings).
  • Wrong: Submit Friday evening (rolling SLA, not weekly cycle).

Self-review limitation — official question (d4-7)

You generate code with Claude then ask the same Claude session to review it. It says the code looks correct. A colleague finds two bugs. Why?

Answer: The model retains reasoning context from generation. In the same session, it is less likely to question its own decisions because it already “knows why” it wrote the code that way. Independent review (different session, no prior reasoning context) is more effective at catching subtle issues.

Exam tip: Any “should we use batch or real-time?” question: blocking = real-time always. Non-blocking = batch candidate. The 50% cost saving is never worth unpredictable latency for blocking workflows.
D4.3 — Few-Shot Examples: When They Help and When They MisleadMEDIUM FREQ
The Rule
Few-shot examples are for teaching decision boundaries and output format. They are NOT for compliance enforcement (use hooks). They work best targeting ambiguous scenarios with explicit reasoning shown. Use AFTER tool descriptions are clear.

When few-shots solve the problem

  • Output format inconsistency: system produces different citation formats for academic papers vs business reports vs government documents. Add few-shot examples showing each document type with the correct, consistent output format.
  • Tool selection for ambiguous queries: after descriptions are expanded, add 4-6 examples targeting the ambiguous cases specifically, showing the reasoning for why one tool was chosen.
  • Escalation calibration: agent escalates straightforward cases while handling complex ones autonomously. Add few-shot examples showing the exact criteria for each decision.

When few-shots do NOT solve the problem

  • Compliance rules: “$500 refund cap” — few-shots improve the model’s tendency but don’t make compliance deterministic. Use a hook instead.
  • Tool ordering: “always call get_customer first” — few-shots show the pattern but don’t guarantee execution order. Use a programmatic prerequisite gate.
  • Before fixing tool descriptions: if descriptions are ambiguous, few-shots are fighting the root cause. Fix descriptions first.
Exam tip: Exam question pattern: “escalation calibration is off” — few-shots with explicit criteria is the correct first step (not a separate classifier model, not sentiment analysis). “Financial compliance rule failing” — few-shots are wrong, use a hook.
D5 — Context & Memory Management
~17% Exam WeightHigh FrequencyD5 Context Management

What it is: Context management addresses how Claude attends to information in large or long-running interactions. Two core failure modes: “Lost in the Middle” (models reliably attend to start and end of long inputs but miss middle content) and Progressive Summarisation Loss (compressing old turns loses numerical values — exact amounts, order IDs, dates — into vague summaries).

Why it matters: In production systems, agents run for many turns, accumulate verbose tool outputs, and handle complex multi-issue sessions. Without explicit context management strategies, accuracy degrades silently — agents reference wrong amounts, miss findings, or produce vague answers referencing “typical patterns” instead of specific findings.

Lost-in-middle fix
Key findings at the TOP of long inputs + explicit section headers throughout. Never “use a larger model” or “increase context window” — those don’t fix attention quality.

Context degradation in long sessions — official question (d5-5)

45 minutes into exploring a large codebase with Claude Code. Claude’s answers become vague, referencing “typical patterns” instead of the specific class names it discovered earlier. What’s happening?

Answer: Context degradation. The context window is filling up and earlier specific findings are being attended to less. Use /compact to reduce context. Maintain scratchpad files recording key findings for reference across context boundaries.

Wrong answers: “The model is rate-limited” (wait and retry), “The codebase is too large” (switch to simpler project), “Temperature is too high” (unrelated).

Aggregate accuracy masking segment failures — official question (d5-6)

Extraction system reports 97% overall accuracy. Manager wants to reduce human review. What should you verify?

Answer: Validate accuracy by document type and field segment, not just in aggregate. A 97% overall rate can mask 60% accuracy on one document type or one field. You must confirm consistent accuracy across all segments before reducing human review.

Exam tip: Summarisation questions: the answer is NEVER “summarise more aggressively” or “adjust summarisation threshold.” The answer is always “persistent case-facts block outside summarisation.”

The Rule
When get_customer returns multiple matches, the agent must ask for additional identifying information. Never select based on heuristics (most recent activity, first result, most orders). Heuristic selection risks acting on the wrong account with financial consequences.

Official exam question (d5-8)

Agent searches for “John Smith” and get_customer returns 4 matches with different account numbers.

OptionVerdictWhy
A: Select most recently active account as most likely matchWRONGHeuristic selection. “Most recent” does not mean “correct.” Risks acting on the wrong account.
B: Ask customer for additional identifying information (email, phone, account number)CORRECTDisambiguation is the only safe approach. Clarification takes one turn; incorrect account takes days to fix.
C: Process request using the first result returnedWRONGArbitrary selection. No basis for assuming first result is correct.

Conflicting statistics from credible sources — official question (d5-7)

Two credible sources: Source A says “market grew 15% in 2024.” Source B says “market grew 22% in 2024.” Synthesis agent must handle this.

Answer: Annotate the conflict with source attribution, presenting both values with their sources and noting the discrepancy. Let the reader assess.

  • Wrong: Average the two values (18.5%). Averaging without basis creates a fabricated number that neither source supports.
  • Wrong: Use the more recent source’s value. Recency does not determine accuracy.
  • Wrong: Omit the statistic entirely. Losing information from both sources is worse than preserving the conflict.
Exam tip: The pattern: when the agent cannot resolve a conflict with confidence, preserve all information with attribution and flag the discrepancy. Do not arbitrarily resolve it. Applies to statistics, tool results, and customer data.
