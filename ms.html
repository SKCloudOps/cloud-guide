01 What is the fundamental difference between High Availability and Disaster Recovery?
⚡ High Freq
▶

Most candidates conflate these two. The distinction is operational continuity vs. existential survival.

High Availability (HA)
Eliminates single points of failure within a running system. Goal: maximize uptime (99.9–99.999%). Operates within one region.
Disaster Recovery (DR)
Restores a system after a catastrophic failure that HA cannot prevent — region outage, data corruption, ransomware. Goal: minimize RTO/RPO.
Masterstroke answer
"HA and DR solve different threat models. HA handles component failure — a node dies, traffic shifts to another. I achieve that with multi-AZ EKS, ALB health checks, and RDS Multi-AZ. But HA cannot protect against an entire AZ or region going dark, or an operator running DROP TABLE on prod. That's DR territory — a deliberate, tested runbook to restore service within agreed RTO/RPO bounds. At Takeda, I separated them explicitly: HA is in the IaC config, DR is a quarterly tested runbook with a pilot-light environment in us-west-2. You need both — HA keeps the lights on daily, DR keeps the business alive after black-swan events."
⚠ What kills candidates
Saying "HA is when you have multiple servers" with no mention of AZ spread, failover mechanics, or the distinction between MTTR and RPO.

03 Define RTO and RPO. Compare all 4 DR strategies.
◈ Deep⚡ High Freq
▶

RTO = Recovery Time Objective — maximum acceptable downtime. RPO = Recovery Point Objective — maximum acceptable data loss measured in time. Lower = more expensive.

StrategyRTORPOCostHow it works
Backup & Restore Hours–Days Hours–Days $ S3 snapshots, RDS backups. Restore from scratch on disaster. Cheapest, slowest.
Pilot Light 10–30 min Minutes $$ Core DB replicating to DR region. Minimal compute off. On disaster, scale up compute and re-point DNS.
Warm Standby Minutes Seconds–Minutes $$$ Scaled-down but running copy in DR region. Handles low traffic. Scale to full on disaster.
Multi-Site Active-Active ~0 (near-zero) ~0 $$$$ Both regions serve production traffic. Route 53 health routing splits load. Global Aurora or DynamoDB Global Tables for data. Failover = DNS weight change.
Masterstroke answer
"The strategy you pick isn't just about RTO/RPO — it's about the business cost of downtime vs. the cost of the DR infrastructure. For Takeda's clinical trial data pipelines, we use Warm Standby because a 4-hour outage could delay trial reporting deadlines worth millions. For internal tooling, Backup & Restore is sufficient. I always start by asking: what's an hour of downtime worth to the business? If it's $50K/hour, Warm Standby or Active-Active pays for itself. If it's $500, Backup & Restore is the right answer."
04 Contrast Monolithic vs. Microservices. Explain Event-Driven Architecture.
◈ Deep
▶

The trap here is being dogmatic. Architects who say "microservices are always better" lose points. Show nuance.

DimensionMonolithMicroservices
DeploymentSingle deployable unit. Fast for small teams.Independent deployments. Enables CI/CD per service.
ScalingScale everything or nothing.Scale only the bottleneck service.
Fault isolationOne bug can crash everything.Failure bounded to one service (with proper circuit breakers).
Operational costLow — one app, one log stream.High — service mesh, distributed tracing, N databases.
Best forEarly-stage products, small teams, well-understood domains.Large teams, diverse scaling needs, mature domain models.
Event-Driven Architecture

Services communicate via events published to a broker (SNS, SQS, EventBridge, Kafka). Producers don't know consumers — loose coupling. Enables: async processing, fan-out, audit trails, temporal decoupling.

Masterstroke answer
"I'd add a third option that's often the right answer: the modular monolith — single deployable, but with strict domain boundaries enforced at the code level. You get monolith simplicity today and an extraction path to microservices when load demands it. At Takeda we run a hybrid: core platform services are microservices on EKS, but some internal tools are deliberately modular monoliths because the team is two people and the operational overhead of 12 services would kill velocity. Event-driven patterns come in where we need async fan-out — e.g., a pipeline completion event on EventBridge triggers S3 archiving, Slack notification, and audit log write in parallel without the pipeline knowing any of those consumers exist."
05 Compare SQL vs. NoSQL from a Solutions Architect perspective.
◈ Deep
▶

Don't recite "SQL has tables, NoSQL has documents." Speak to access patterns, consistency models, and scaling limits.

FactorSQL (RDS/Aurora)NoSQL (DynamoDB)

"The architect's decision is driven by access pattern analysis, not data shape. If the team will write ad-hoc queries and the schema will evolve, SQL wins — flexibility matters more than performance. If the application has 3 known access patterns and needs to scale from day one, DynamoDB wins — you pre-bake the patterns into the table design with partition and sort keys. The anti-pattern I've seen repeatedly: teams start with DynamoDB because it's 'serverless and scalable,' then discover at 6 months they need a join and they're stuck. Know your query patterns before choosing the database."
06 How would you architect a fully available and fault-tolerant application on AWS?
⚡ High Freq◎ Scenario
▶

Availability = % of time service is accessible. Fault tolerance = ability to continue operating despite failures. Fault tolerant is a stronger guarantee.

1
Eliminate single points of failure at every layer. Multi-AZ deployment for compute (EKS node groups across 3 AZs), data (Aurora Multi-AZ), and networking (NAT Gateway per AZ, not shared).
2
Design for graceful degradation. Circuit breakers (AWS App Mesh / Istio) prevent cascade failures. Read from cache (ElastiCache) when DB is degraded. Return stale data rather than 500 errors.
3
Decouple with async patterns. SQS queues absorb traffic spikes and decouple producers from consumers. A downstream service outage doesn't propagate upstream.
4
Immutable infrastructure + rapid replacement. EC2 instances are cattle, not pets. ASGs replace unhealthy nodes automatically. EKS self-heals pods. Recovery time = seconds, not hours.
5
Health checks everywhere. ALB target health checks, EKS liveness/readiness probes, Route 53 health checks for cross-region DNS failover.
6
Chaos engineering in staging. AWS Fault Injection Simulator. Validate that the theory matches reality before production.
Masterstroke closer
"The difference between availability and fault tolerance becomes real under fire: an available system recovers from failure. A fault-tolerant system never stops serving — it handles the failure in-flight. For critical workloads at Takeda, I target fault tolerance for the read path (always return something) and high availability for the write path (recover within 30 seconds). That asymmetry dramatically simplifies the architecture while meeting SLAs."

11 Your entire production workload runs in a single AWS region — design a multi-region DR strategy with minimal downtime.
◎ Scenario◈ Deep
▶

This is a design-under-constraints question. Show you understand the phased migration approach rather than jumping to Active-Active.

1
Define RTO/RPO first. The answer changes completely based on whether the business can tolerate 1 hour vs. 5 minutes of data loss. Get these numbers before designing.
2
Start with Pilot Light (phase 1). Enable Aurora Global Database cross-region replication — <1s lag. Deploy minimal compute in DR region via existing Terraform. Cost: ~15% of primary.
3
Add Route 53 health checks. Configure failover routing policy: primary = us-east-1, secondary = us-west-2. Route 53 checks ALB health every 30s. Automatic DNS failover within 60–90s of primary failure.
4
Automate the failover runbook. Lambda function triggered by Route 53 health alarm: promotes Aurora global secondary to primary, scales up DR EKS node groups, updates any region-specific configs. Target: <10 minute RTO automated.
5
Test quarterly via GameDay. Actual DNS failover to DR region with synthetic traffic. Measure actual RTO vs. target.
Masterstroke answer
"The trap is building Active-Active before you need it — it's expensive and operationally complex. I'd implement Pilot Light as the first milestone: Aurora Global Tables for near-zero RPO, Route 53 health-check failover for automated RTO under 10 minutes. That's 80% of the benefit at 20% of the cost. If after 6 months the business validates it needs RTO under 60 seconds, then we invest in Warm Standby or Active-Active. The other thing I'd insist on: the runbook must be code, not a document. A Lambda-based automated failover tested monthly beats a 40-page PDF that no one has read."

02 How to design for both regions active at once with separate access control?
◈ Deep◎ Scenario
▶

This is Active-Active with data sovereignty or access control separation — common in regulated industries or global products.

1
Traffic routing: Route 53 Geolocation or Latency routing — EU users → eu-west-1, US users → us-east-1. Weighted routing for canary across regions.
2
Data layer: DynamoDB Global Tables or Aurora Global Database for active-active data replication. Conflict resolution strategy: "last writer wins" for DynamoDB, application-level for Aurora.
3
Access control separation: Separate AWS accounts per region under AWS Organizations. IAM roles scoped per account — EU team cannot assume roles in US account. SCPs at OU level enforce data residency (e.g., deny actions that replicate data outside EU).

"For GDPR compliance specifically, I'd add an SCP: Deny s3:ReplicateObject if destination bucket is outside eu-west-1. The SCP sits above IAM in the policy evaluation hierarchy, so it's enforced regardless of what the application or developer does."

06 Share AMI with child accounts? Shared AMI instance launches and terminates? Diagnose shared AMI issues?
◎ Scenario
▶

Three connected questions — answer them as a single flow showing you understand the AMI + snapshot permission model.

Sharing an AMI
aws ec2 modify-image-attribute \ --image-id ami-0abc123 \ --launch-permission "Add=[{UserId=CHILD_ACCOUNT_ID}]" # CRITICAL: Also share the underlying EBS snapshot aws ec2 modify-snapshot-attribute \ --snapshot-id snap-0abc123 \ --attribute createVolumePermission \ --operation-type add \ --user-ids CHILD_ACCOUNT_ID
Why shared AMI instance launches then terminates
  • Missing snapshot permission — AMI shared but underlying EBS snapshot not shared. Instance fails to attach root volume → immediate termination. Most common cause.
  • AMI uses encrypted snapshot with KMS key not shared to child account. Decrypt fails silently → terminate.
  • Instance profile in child account lacks permissions required by user-data script at boot.
  • Target instance type not available in child account's region (e.g., newer family not enabled).
Diagnosis steps
  • Check EC2 System Log: aws ec2 get-console-output --instance-id i-xxx
  • Verify snapshot is shared: aws ec2 describe-snapshot-attribute --snapshot-id snap-xxx --attribute createVolumePermission
  • Check KMS key policy for cross-account grants if AMI is encrypted.
  • Review CloudTrail in child account for RunInstances → look for error codes.
Masterstroke answer
"The instant-terminate scenario is 9 times out of 10 a missing snapshot permission. The AMI is visible to the child account but the actual EBS snapshot backing it isn't shared — so when EC2 tries to create the root volume, it gets an access denied, and the instance terminates. The fix is two operations: share the AMI and share each snapshot in the AMI's block device mapping. For encrypted AMIs, you also need to grant the child account's service role kms:Decrypt and kms:CreateGrant on the KMS key. I've encoded this as a Terraform module that handles all three permissions atomically — you can't share the AMI without also correctly configuring the snapshot and KMS permissions."

19 Users report intermittent failures across microservices, but logs from individual services look normal. How do you identify root cause?

02 Internet Gateway is attached to a subnet or a VPC?
⚡ High Freq
▶

Classic trick question. The answer is VPC, not subnet.

Masterstroke answer
"The IGW is attached to the VPC — one IGW per VPC, and it's horizontally scaled and redundant by AWS. 03 Where are the outgoing rules for a private subnet needing internet access?
⚡ High Freq
▶

Tests whether you understand the NAT Gateway placement model and routing asymmetry.

Masterstroke answer
"Two route tables work together: the private subnet's route table has 0.0.0.0/0 → NAT Gateway. The NAT Gateway itself lives in a public subnet whose route table has 0.0.0.0/0 → IGW. So outbound internet traffic from a private instance flows: instance → NAT GW (source NAT applied, private IP replaced with NAT GW's Elastic IP) → IGW → internet. 05 Why do you need Transit Gateway when we have VPC Peering? How is TGW beneficial?
⚡ High Freq◈ Deep
▶

This tests your understanding of N² peering problem and operational scalability.

VPC Peering
1:1 connection between 2 VPCs. Not transitive — if A peers with B and B peers with C, A cannot reach C. N VPCs = N*(N-1)/2 peering connections. 10 VPCs = 45 connections.
Transit Gateway
Hub-and-spoke model. N VPCs connect to 1 TGW = N connections total. Transitive routing. Supports VPN, DX, peering between TGWs across regions/accounts.
When VPC Peering is still better
  • 2–3 VPCs only, simple routing — VPC peering is free (no hourly charge), TGW costs ~$0.05/hour per attachment + data processing fee.
  • Cross-account peering where you want explicit, minimal blast radius connectivity.
  • When you need lower latency — TGW adds a small routing hop.
Masterstroke answer
"VPC Peering breaks at scale for two reasons: the N² connection problem and non-transitivity. At Takeda we have 15+ VPCs — managing 100+ peering connections, each with its own route table entries and security group rules, is operationally impossible. Transit Gateway collapses that to 15 attachments and a single routing domain. Beyond that, TGW adds capabilities peering never had: multicast support, centralized egress to a shared NAT Gateway, inter-region peering (TGW peering across regions), and a single place to attach Direct Connect via a Direct Connect Gateway. The cost is real — TGW has hourly + data charges — but at 10+ VPCs the operational savings far outweigh it."

One of the most important security questions. The evaluation order has a strict priority hierarchy and explicit deny always wins.

1
Explicit DENY anywhere = DENY. Full stop. No other policy can override an explicit deny. Check SCP, resource policy, identity policy, permission boundary — any explicit deny ends evaluation immediately.
2
Service Control Policy (SCP) — Must have an Allow in SCP. SCPs don't grant permissions — they set the maximum permissions boundary for an account. Even if all other policies allow, an SCP that doesn't allow the action = DENY.
3
Resource-based policy — If the resource (S3 bucket, KMS key, SQS queue) has a resource policy that allows the principal, access is granted even without an identity policy (for same-account). Cross-account requires BOTH resource policy AND identity policy to allow.
4
Permission boundary — Sets the maximum permissions for an IAM entity. The effective permission is the intersection of identity policies AND permission boundary.
5
Identity-based policy — Inline or managed policies attached to the user/role. Must have an explicit Allow here.
6
Session policy — For assumed roles, the session policy (passed at assume-role time) further restricts permissions. Cannot grant beyond the role's identity policies.
Masterstroke answer
"The mantra is: default deny, explicit deny wins, SCPs set the ceiling, permission boundaries set the per-identity ceiling, then identity policies grant within all those constraints. The cross-account case is the most common source of bugs: a developer adds an S3 bucket policy allowing Account B, but the IAM role in Account B doesn't have s3:GetObject in its identity policy. Access denied — both sides must allow for cross-account. Same-account is different: a resource policy allowing a same-account principal is sufficient without an identity policy, because the resource policy implicitly trusts the account."

14 How does AWS STS work internally? How do you implement cross-account access securely?
◈ Deep

STS is the temporary credential vending service — everything in AWS that needs short-lived credentials goes through STS.

How STS works
  • You call sts:AssumeRole with the target role ARN and an optional external ID and session policy.
  • STS verifies your identity, checks the role's trust policy to confirm you're an allowed principal, and validates SCPs/permission boundaries.
  • Returns a temporary set of credentials: AccessKeyId, SecretAccessKey, SessionToken — valid for 15 min to 12 hours.
  • The temporary credentials are scoped to the assumed role's permissions, further limited by any session policy passed at assume time.

15 What is AWS Organizations and SCP? How do Service Control Policies work?
⚡ High Freq◈ Deep
▶

SCPs are the guardrails that sit above IAM — they constrain what even Account Admins can do.

  • SCPs apply to all IAM principals in member accounts (including root user, except for management account).

20 What would you do if Amazon EKS runs out of IP addresses?
◎ Scenario◈ Deep
▶

EKS with VPC CNI assigns each pod a real VPC IP from the node's subnet. IP exhaustion is a real operational risk in EKS.

Why it happens
  • VPC CNI pre-allocates IPs to each node (warm IP pool). A node with 8 pods might hold 12 reserved IPs. IP consumption is higher than pod count suggests.
  • Subnets created with small CIDRs (/24 = 251 usable IPs, minus 5 reserved = 246) fill up with large node counts.
Solutions
1
Immediate: Enable VPC CNI prefix delegation — each node gets a /28 prefix (16 IPs) per ENI instead of individual IPs. Increases pod density per node significantly without new subnets.
2
Add secondary CIDR: Associate a secondary CIDR block (e.g., 100.64.0.0/16 — RFC 6598 range) to the VPC. Create new /24 subnets from this range. Assign these as node subnets. Pods get IPs from the secondary range, freeing primary CIDR for other resources.
3
Custom networking: VPC CNI custom networking mode — pods get IPs from a secondary subnet different from the node's primary subnet. Decouples pod IP space from node IP space.
4
Alternative CNI: Cilium in overlay mode — pods get IPs from an internal CIDR not tied to VPC. No VPC IP consumption per pod. Requires replacing VPC CNI entirely.
Masterstroke answer
"The fastest fix with no architecture change: enable prefix delegation on the VPC CNI. One command: kubectl set env daemonset aws-node -n kube-system ENABLE_PREFIX_DELEGATION=true. This immediately increases available IPs per node by using /28 prefixes. For long-term: always create EKS node subnets at /22 or larger and use the secondary CIDR (100.64.0.0/16) strategy from day one. I've built this into Takeda's EKS Terraform module as a default — new clusters provision /22 node subnets and enable prefix delegation by default, so we never hit this problem again."

33 How do you ensure zero downtime deployments?
⚡ High Freq

Pod Intent / Desired State

Kubernetes is a declarative desired-state reconciliation system. You declare intent (3 replicas of myapp). The control loop continuously measures actual state and drives toward desired. If a node dies and takes 2 pods with it, the controller notices actual (1) ≠ desired (3) and schedules 2 new pods. This reconciliation loop is what "self-healing" means concretely — it's not magic, it's a control loop running every few seconds.

01 How can you make your application scalable for a big traffic day?
⚡ High Freq◎ Scenario
▶

6

7
